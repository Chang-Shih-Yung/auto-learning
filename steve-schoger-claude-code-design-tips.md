# Steve Schoger × Claude Code：設計師的工作流程拆解

> Steve Schoger（Tailwind Labs 設計師）發布了一支一小時的實錄影片，展示他如何用 Claude Code 從零建出一個金融 App 的行銷首頁。這份筆記聚焦在他的**工作方式與過程**。

---

## 他是誰，背景是什麼

Steve Schoger 是 Tailwind Labs 的設計師，與 Adam Wathan 長期合作，也是 *Refactoring UI* 的共同作者。他目前在開發 **UI.SH**，一套讓 Claude Code、Cursor、Amp 這類 AI agent 在寫前端時能套用專業設計標準的 skills 套件。

---

## 工作環境

| 項目 | 說明 |
|------|------|
| 技術棧 | Vite + Tailwind CSS + React |
| 模板 | 由 Adam Wathan 協助建好的 Vite 模板，開新專案直接複製 |
| 畫面配置 | 左側 localhost 即時預覽，右側 Claude Code 終端 |
| Figma | **只剩做向量圖形（SVG logo）**，其餘全部在 Claude Code 裡完成 |

他自嘲說自己只會兩件事：`cd`（換目錄）和啟動 Claude。

> 「我對命令列還是非常新手，這些東西對我來說都很陌生。」

---

## 這次影片的過程

1. **從空白畫面開始** — 不用任何 skills 或 CLAUDE.md
2. **一個起手 prompt** 描述整個頁面結構，Claude 生成初版
3. **約 50 次迭代對話**，逐步從「AI 預設輸出」調整成專業成品
4. 全程沒有寫任何程式碼

### 起手 Prompt 長什麼樣

> 「幫一個金融 app 設計一個簡單的行銷首頁，這個 app 把所有收入來源整合到一個 dashboard 裡。頁面要包含導覽列、hero 區塊、logo 列、功能介紹、數據統計、使用者推薦、CTA 區塊和頁尾。」

重點是：不寫程式碼、不指定 CSS 屬性，**用設計語言描述結構與意圖**就夠了。

---

## 他怎麼跟 Claude 說話

Schoger 的 prompt 風格跟工程師完全不同，用的是**設計師的語言**：

| 類型 | 範例 |
|------|------|
| 具體數值 | 「改成 38 像素」「改成 gray-950」 |
| 模糊方向 | 「再大一點點」「再緊一點」 |
| 設計描述 | 「加一個外側的淡色邊框取代這個 solid border」 |

> 「我覺得我寫的這些句子根本不太合理，但它每次都聽得懂。」

### 三個特別有效的 prompt 模式

**1. 問「這個是怎麼做的？」**
先讓 Claude 解釋目前的實作方式，確認問題所在後再精確指出要怎麼改。比直接說「改一下」更有效，因為你知道你在改什麼。

**2. 全站同步**
調好一個 section 的樣式後，說「把這個樣式套用到下面所有區塊」，一次統一風格，不用逐一修改。

**3. 建臨時工具，用完就丟**
他讓 Claude 做了一個**拖曳定位工具**，讓他可以視覺化地調整截圖位置，調好之後複製座標值寫進程式碼，再叫 Claude 把工具移除。把 Claude 當臨時腳手架用。

## Claude 的自動學習行為

Schoger 注意到一件事：他在前面的對話中定義過的樣式（例如按鈕的 ring 外框），Claude 在後面新增元素時會**自動沿用**，不需要他再說一次。

> 「它就是一邊做一邊學，這還蠻好的，我不需要再額外更新。」

這表示越早建立視覺規範，後面的迭代就越省力。

---

## 迭代過程的節奏

50 次對話不是隨機的，有一個大致的順序：

```
起手 prompt → Claude 生成初版（AI 預設風格）
    ↓
第一波：打破 AI 預設（對齊、顏色、字型）
    ↓
第二波：細化元素（邊框、陰影、按鈕、容器）
    ↓
第三波：排版細節（tracking、行高、max-width）
    ↓
第四波：裝飾收尾（Canvas Grid、testimonial、logo cloud）
```

每次迭代都是**一個問題、一個指令**，不會一次要求改很多東西。

---

## AI 初版 vs 設計師調整後的差距

這是整個影片最核心的示範。同一個 prompt，AI 生成的初版和經過 50 次迭代的成品，**看起來完全不同**。

| AI 的預設行為 | 設計師的調整方向 |
|--------------|----------------|
| 用 indigo 當主色 | 根據品牌調整配色 |
| 所有東西置中對齊 | Hero 改左對齊、非對稱佈局 |
| solid border + shadow（混濁感） | 改用 outer ring（10% opacity） |
| 所有圓角用同一個值 | 同心圓角（內層 = 外層 − padding） |
| Google Fonts 固定字重 | Inter variable font 中間字重（如 550） |
| 置中的 section heading | 標題與副標同行，不同顏色與字重 |

**差距不在程式碼，在設計判斷。**

---

## 這個工作流的本質

以前的設計流程：
```
設計判斷 → Figma 繪製 → 交給工程師 → 手寫 CSS
```

Schoger 現在的流程：
```
設計判斷 → 用自然語言告訴 Claude → 直接看到結果
```

設計師的核心價值（眼光、判斷、品味）沒有被取代，**執行層的障礙被移除了**。他沒有寫任何一行 CSS，卻做出了可以拿出來說嘴的專業成品。

這也是他做 UI.SH 的原因：**把這套設計判斷打包進去，讓 AI 從一開始就不走彎路。**
