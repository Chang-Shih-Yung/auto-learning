## WUPHF 826★：多 agent 共享辦公室，每輪 87k tokens 平的、五輪 Claude Code 0.06 美元

你可以在自己的 Mac 上跑 wuphf，開 `localhost:7891` 看到一個共用的 web 工作空間，裡面同時躺著 Claude Code、Codex、OpenClaw、本地 LLM——每個 agent 各自有自己的 notebook（私有），共同寫一份 team wiki（公開、git 追蹤），協作流程全部肉眼可見。實測數字：Codex 走完一個 10 輪 CEO session，整個 office 累計只花約 286k tokens（cache 後）；每輪 input 平的維持在 ~87k，相比之下傳統 multi-agent orchestrator 跑到第 8 輪就累積到 484k；Claude Code 5 輪對話只花 0.06 美元；idle 時 token burn 是零。

這幾個數字背後有兩個結構決定。一是 **session model**：每輪都從 scratch 起來、不累積歷史，靠 wiki + notebook 持久化決策結果，而不是把 turn-by-turn message 全塞進 context——這直接打到 Anthropic prompt cache 對齊規範，達成 97% cache hit rate。二是 **broker 不輪詢**：Go 寫的 broker 用 push 觸發 agent，agent 沒事就睡，不會像傳統多 agent 框架那樣靠 heartbeat polling 燒 token。Memory backend 是 markdown + git，不是 vector DB——意思是你可以對 wiki 直接 `grep`、用 `git log` 看誰改了什麼、把整份 team memory `git clone` 帶走；MCP tools 也按 mode 動態 scope，DM 模式只 4 個工具、full office 模式才開到 27 個，把模型每次思考時的 cognitive surface area 壓到最小。

以前 multi-agent 系統的痛點是 token bloat——每個 agent 都要看完整對話歷史、orchestrator 把 8 輪訊息壓到一條 prompt 灌進來、idle 時還在 polling。現在 wuphf 用三個動作把這些都歸零：歷史進 wiki（不在 context）、idle 用 push（不再 polling）、tool scope 按情境收（不一次給滿）。對你這個正在 Nexus 寫 MCP server、又想把多種 AI client 串起來的人，這份原始碼就是「Memory Bundle = 程序 + 情節 + 因果同時載入」這個你早就在想的設計，落地成一個 826 star 的可跑專案——而且記憶層用的是 markdown + git，跟你既有的設計 token 工作流一個世界觀。

## VS Code 把 `Co-authored-by: Copilot` 默默注入你的 commit，PR #310226 引爆 697 點 HN

你打開 VS Code 1.106 之後手寫一個 commit message、按下 commit，回頭 `git log` 卻發現多了 `Co-authored-by: Copilot <copilot@github.com>` 這行 trailer——即使你完全沒呼叫 Copilot、即使你在 settings 裡設了 `chat.disableAIFeatures: true`。這是 PR #310226 把 `git.addAICoAuthor` 設定的預設值從 `"off"` 改成 `"all"` 引發的 regression，HN 上累積到 697 點、318 則討論，Microsoft 把整串鎖了標為 spam，維護者承諾在 1.119 修。要立刻避開：在 settings.json 加 `"git.addAICoAuthor": "off"`，已經污染的 commit 用 `git rebase -i` + `git commit --amend` 把 trailer 拔掉。

技術上的修補只動了兩行——`extensions/git/package.json` 跟 `extensions/git/src/repository.ts` 把 schema default 跟 runtime fallback 都對齊到 `"all"`。但這個 bug 暴露的不只是預設值錯——是 **VS Code 沒有可靠機制判斷某段 code 實際上有沒有用 AI 生成**。預設打開等於「除非你關掉，否則一律假設你用了 AI」。維護者 dmitrivMS 自己承認：「It should never have been enabled when `disableAIFeatures` is on，It should not add attribution to changes that were not done by AI」——兩個 should，沒有一個現在做得到。HN 評論裡有人懷疑這是為了在財報上把 Copilot adoption metric 灌水，因為被掛 `Co-authored-by: Copilot` 的 commit 就會被 GitHub Insights 計入「Copilot-assisted commits」。

以前 commit 的 trailer 是 commit author 主動加的，是個信任訊號——`Co-Authored-By` 過去意味著「真的有 pair programming」。現在 IDE 默默幫你掛名共同作者，這個訊號崩壞了。對你這個跑 Claude_auto 每天自動 commit、又用 `Co-Authored-By: Claude` trailer 標記 AI 介入的人，這不只是 VS Code 的 regression——是更大的問題：**「AI 介入率」這個 metric 不能由用 AI 的工具自己回報，必須由獨立的稽核層量測**。如果你之後做 Nexus 的合作署名稽核或 AI 工作流分析，VS Code 注入的 trailer 是不可信的訊號源，得跟你自家的 git hook（記錄真正的 Claude Code 呼叫時點）交叉比對才能用。

## Context Gateway 596★：把 grep 跟 read 的雜訊在 proxy 層用 SLM 抽掉，< 0.5 秒延遲

你可以在 Claude Code 跟 Anthropic API 之間架一層 Apache-2.0 的 Context Gateway proxy（596 stars），它會在背景監測你的 context 用量——一達到 window capacity 的 85% 就在後台跑壓縮，使用者感覺像沒壓縮過，每次只多 < 0.5 秒延遲，而且因為 prefill 的 context 變短、speedup 反而把這個延遲補回來。壓縮目標主要不是你的 prompt，而是 tool output——`grep`、`read`、`ls` 動輒幾千 tokens 的雜訊，先過一個小的 SLM classifier 過濾掉跟當前 tool call intent 無關的部分再進主模型。同樣的架構支援 Claude Code、Cursor、OpenClaw、自訂 agent，自帶 summarizer model 可選。

設計核心是兩個觀察接起來。一是 **long context 是有代價的**——OpenAI 自己的 GPT-5.4 評測顯示，accuracy 從 32k tokens 的 97.2% 一路掉到 1M tokens 的 36.6%，模型不是 context 越大越好，是越大越笨。二是 **Claude Code 內建的 `/compact` 太晚才跑**——它是同步阻塞的，等你 context 塞滿了才觸發、用戶看著螢幕等好幾秒。Context Gateway 把這個流程改成「pre-compact at 85%」、在背景非同步跑，並把壓縮顆粒度從「整段歷史」細化到「單個 tool output」——因為大多數 token 浪費就在 grep 一次回傳 5,000 行 build log 這種地方。HN 上 sethcronin 質疑「中間人壓縮會不會把 agent 真正需要的線索剪掉」，作者回應：「不壓縮也是風險，context 一過大模型就不會用了，所以解法是壓縮但別剪掉 signal」——這就是 SLM classifier 要解決的事，不是隨機砍。

以前 Claude Code 的 context 管理是兩個極端：要嘛你眼睜睜看 token 暴漲、模型開始亂寫；要嘛你按 `/compact` 等同步壓縮把 session 凍結幾秒。現在多了第三條路——把 context 治理當成獨立的 proxy 層，用 SLM 做粗篩、預壓縮在背景跑。對你這個每天用 Claude Code 寫 Nexus 跟 Claude_auto、跑長 session 跑到 `/compact` 卡住過的人，這就是把昨天那條洞察「context 不是用來存歷史的，是用來做決策的」實作成一個你可以直接架在工作流之前的中介層——proxy 不是必要的，但它揭示了一種可能：**context 治理是基礎建設，不是模型本身的責任**。

## Pu.sh：396 行 shell 實作完整 coding agent harness，37KB vs Claude Code 209MB

一個叫 NahimNasser/pu 的開源專案，用 396 行 bash + curl + awk 寫了一份完整的 coding agent harness——包含 7 個內建工具（bash / read / write / edit / grep / find / ls）、interactive REPL with multi-turn memory、context auto-compact when budget exceeded（OpenAI 約 400KB context、Opus 約 272KB）、checkpoint/resume via `.pu-history.json`、session export to markdown、JSONL event logging、pipe mode 讓多個 agent 串起來、surgical text replacement 保留 trailing newline 的編輯。MIT license，自動偵測 API key 路由到 Anthropic 或 OpenAI（內含 Opus 4.7 與 GPT-5.5 支援），預設走 Anthropic。檔案大小放在尺度上才看得出意義：pu.sh 37KB / 396 lines，Claude Code 209MB、Goose CLI 237MB、Pi+Node 281MB、SWE-agent Docker 鏡像 1.8GB——同樣的「coding agent harness」抽象，可以從 1.8GB 縮到 37KB，差了 5 萬倍。

技術上 pu.sh 沒有任何 magic。它是把 LLM tool-use 的本體拆解成最小骨架：把 system prompt 寫死在 shell 變數、tool call 用 awk 解析 JSON 回傳、context 超過 budget 就用 awk 做 sliding window 截掉前段、persist 用 `cat >> .pu-history.json` 寫 JSONL。沒有 npm dep tree、沒有 Python venv、沒有 Docker 鏡像——只有一份 shell + curl + awk 跑得起來的環境就能跑。設計者自己寫了 30+ 個實驗逼這個答案：「最 portable 的 agentic harness 形態是什麼？」結論不是 Rust binary、不是 Go binary——是一個 shell script，因為 shell 是所有系統都有的。這條哲學路徑不是要取代 Claude Code（它不能、也不打算），是揭示「coding agent 的本體最少需要多少」。

以前要把 coding agent 嵌進 GitHub Actions runner、Lambda、邊緣 IoT 設備、tmux pane、CI container、學校教室的舊 Mac，光是裝 Node + npm + 跨平台 dep 就要 200MB+；現在 → 一個 `curl` 就把 37KB 的 shell script 拉下來直接跑，連 npm install 都跳過。對你這個正在 Nexus 寫 CLI、又對「先 CLI 後 MCP」順序感興趣的人，pu.sh 就是 portability 哲學推到極致的範例——當你哪天想把 Nexus 的設計 token check 包成「在任何 GitHub Actions runner 上跑、不裝 Node 也能用」的 agent 時，這個架構就是參考點。它不會取代你的 Nexus CLI，但它告訴你：當 portability 是首要約束，shell 永遠贏 Node。

## 今日總結

今天四篇文章串起來指向同一個結論：**AI agent 的工程瓶頸已經從「模型多強」徹底搬到「context infra 怎麼設計」**。

**WUPHF 826★** 把 multi-agent 共享記憶搬到 git-tracked markdown，每輪 87k 平的（vs 業界 484k 累積）、idle 0 token burn——示範「不用 vector DB、用你已會的 git」這條路。**Context Gateway 596★** 把 SLM classifier 架在 proxy 層，pre-compact at 85% capacity、< 0.5 秒延遲——示範「context 治理是基礎建設、不是模型責任」。**Pu.sh 396 行 shell** 把整個 coding agent harness 縮到 37KB（vs Claude Code 209MB、SWE-agent 1.8GB）——示範「最 portable 的 agent 形態是 shell」。三條獨立的線索，三個不同的設計選擇，同一個共識：**模型已夠強，context 怎麼餵、怎麼壓、怎麼搬，才是工程題**。

**VS Code Copilot Co-author auto-inject (PR #310226, 697 點)** 是這個共識的反面教材——當你把 AI 介入率的量測權交給「使用 AI 的工具自己」，得到的訊號就會被默默污染（即使 disableAIFeatures=true，commit trailer 還是會被掛 Copilot）。Microsoft 鎖討論串、承諾 1.119 修，但揭示了更根本的設計題：**AI workflow 的稽核訊號必須由獨立稽核層量測，不能由執行層自我回報**。

對 Henry 自己的應用點：
1. **Nexus 設計 token skill 該開始量測 capability / trigger / cost 三軸**（接續 5/2 NgRx Signals Skill 的洞察），WUPHF 的 wiki backend 可以當這套量測結果的 git-tracked 持久層
2. **Claude_auto 的 commit trailer 不能依賴 VS Code 預設值**——加自家 git hook 量測真實 AI 介入率，跟 IDE 注入的 trailer 交叉驗證
3. **portability 框架可從 Pu.sh 借**——把 Nexus 的 sub-skill（如 token contrast check）抽成 shell-only agent，零依賴跑在任何 CI runner
