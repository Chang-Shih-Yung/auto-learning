## Claude Code v2.1.126：`/model` 改吃 gateway 的 `/v1/models`，OAuth 終於能在 SSH/WSL2 直接貼 code

你可以在 v2.1.126 把 `ANTHROPIC_BASE_URL` 指到任何 Anthropic-compatible gateway（自架 proxy、企業內部 LiteLLM、本地 Ollama 中繼），按下 `/model` 就會直接列出該 gateway 的 `/v1/models` 端點上有的所有模型——不用再手動編 `settings.json` 把模型名稱硬塞。同一版加了 `claude project purge` 命令來清掉所有 Claude Code 專案級狀態，支援 `--dry-run` 預覽和互動模式選擇要砍哪些；OAuth 登入流程現在接受直接在 terminal 貼授權碼，WSL2、SSH session、container 環境再也不用先把瀏覽器導到本機才能登入。Windows 端把 PowerShell 偵測到時直接視為主 shell；deferred tools 跟 plan-mode tools 在 subagent 與互動 session 都能正常使用的 bug 也修了。

這版的設計哲學接續 v2.1.116 的 deferred loading 與 v2.1.121 的 alwaysLoad——把工具定義權還給 server，把模型清單權還給 gateway。以前 `/model` 是 Claude Code 自己 hardcode 的清單，你跑自架 proxy 時要嘛改原始碼、要嘛在 settings 裡塞死字串；現在 Claude Code 變成「對齊 OpenAI 規格」的客戶端，`/v1/models` 拿什麼就顯示什麼。OAuth 終端貼 code 看似小事，但把「無頭環境的登入流程從『先 port forward』降級成『複製貼上』」就是把摩擦歸零的典型例子——SSH 進開發機、開個 dev container、WSL2 跑 CLI 全部不用再開瀏覽器繞一圈。

以前要在 SSH 主機跑 Claude Code，登入就是個 30 秒的 port forwarding 折磨；要切到自架 gateway 上的 GPT-5.5 或本地 Qwen，要手動編 `settings.json` 才看得到那個模型。現在，gateway 對齊 `/v1/models`、OAuth 接受貼 code、`claude project purge` 把專案狀態當作可重置的物件——三個獨立修補加起來，就是把 Claude Code 從「桌面為中心」轉成「能在任何 shell 環境裡跟你共存」的工具。風扇不狂轉、proxy 不爆設定、SSH session 不卡 callback。

## NgRx Signals 的 Anthropic Skill：5 個任務、41 條斷言，通過率從 84% 升到 100%

你可以用 Daniel Sogl 開源的這份評測腳本，把任何一份 Anthropic Agent Skill 跑一輪量化測試——他用 NgRx Signals 為例，設計 5 個真實任務（withState 型別宣告、optimistic update + rollback、entity filtering、updateEntity pattern、standalone updater），合計 41 條斷言，跑兩次：載入 skill、不載入 skill。結果是通過率 100% vs 84%（+16pp），代價是每次呼叫多 13.7 秒、多 12,416 個 token、多花約 0.04 美元。同一套架構還跑了 trigger optimization：20 個合成 prompt（10 個該觸發、10 個不該觸發）做 train/test split，自動跑了三輪重寫 description，結果 recall 都卡在 42.9% 上不去——跟 Vercel 公開過的「skill 在 56% 該觸發場景被忽略」吻合。

這套評測值得抄是因為它把「skill 寫得好不好」這件原本靠手感的事，拆成三個可量測的軸：capability（會不會做對）、trigger（該觸發時會不會觸發）、cost（每次多花多少）。`withState<CartState>()` 顯式型別、`getState()` 拍快照給 rollback 用、`updateEntity` 的 predicate form——這些 NgRx Signals 的 idiom 不寫進 skill，模型 84% 場合會做出某個語法看起來對但 runtime 行為錯的版本。寫進 skill 之後，41 條斷言全綠，但 trigger plateau 揭示了真正的瓶頸：skill 描述再怎麼改，模型在某些場合就是不會主動載入，這已經不是 prompt engineering 的問題，是 skill 註冊與選擇機制本身的設計題。

以前你寫 Claude Code skill 或 Anthropic Agent skill，只能靠「我跑了幾次感覺對」當驗收標準；遇到 skill 沒被載入，只會以為自己 description 寫得太爛。現在有了一份可複製貼上的評測腳本——5 個任務 + 41 條斷言 + 20 個 trigger prompt——把 skill 質量變成一個 CI 步驟。對 Henry 這種同時在 Nexus 寫設計 token skill、又想把這條路徑當成 AI workflow 主軸的人，這就是把「skill 是 markdown + 希望」翻譯成「skill 是有 SLA 的工程產物」的範例。

## 「為什麼選 CLI 不選 MCP」：當你工具鏈在 Claude Code、Kimi、Cursor 之間切

你開一個 npm 套件叫 `experiwall-cli`，跑 `npx experiwall init`、再跑 `npx experiwall test`——同樣這個 A/B testing 工具如果走 MCP 路線，你要先在每個 agent 平台註冊 server、宣告 schema、處理 platform-specific 設定，每換一個 agent 就要重做一次。Marcelo Ricigliano 寫了一篇從實戰得到的對照：他自己同個工具同時提供 CLI 與 MCP 兩條路徑，發現使用者真正在用的幾乎都是 CLI——因為 CLI 是 OS 層的，任何能跑 shell 的 agent 都直接相容；MCP 是整合層的，被綁在每個 agent 自己的 schema 註冊機制裡。

兩條路徑的真實 trade-off 是「token 預算花在哪」。MCP 在 agent 裡會把 schema 灌進 context，模型每次思考都要評估「這個 tool 是不是現在該用」——多一個 MCP 就多一塊 cognitive surface area。CLI 反過來把 schema 留在 OS 層，agent 只要會讀 `--help`、會解析 stdout，就能呼叫；模型不需要把 schema 放進記憶。MCP 不是沒優勢——回傳結構化 JSON 給模型直接吃，省掉「解析 terminal 雜訊」這一步；但這個優勢要在工具回傳資料量大、結構複雜時才划算，小工具放 MCP 等於用 token 買沒人需要的整潔。

以前的設計慣例是：要做 AI 整合就上 MCP，因為新潮、結構化、官方推。現在的真實使用形態揭露了一條更務實的路：先寫 CLI，因為它跨 agent 平台不用重做、不吃 context token、靠 OS 寫一次到處跑；只有當回傳資料真的需要結構化、agent 真的反覆處理時，才升級成 MCP。對 Henry 這種已經在 Nexus 寫過 MCP server、又每天在 Claude Code 跟其他 AI client 之間切的人，這個「先 CLI 後 MCP」的順序就是把工具設計從「最潮的」改成「最 portable 的」。

## 28 對 query 拆解 semantic cache：embedding 模型沒在說謊，真正的失敗模式跟你想的相反

你可以用 `text-embedding-3-small` 跑 28 對 query，量化兩種容易出包的情境：entity swap（同樣問句、換掉資料庫名稱，例如 Dragonfly → Valkey），以及 paraphrase（同樣意圖、換句話講）。Kristian Ivanov 的實測結果是 entity swap 的 cosine similarity 落在 0.15–0.74（最高 0.7391），paraphrase 落在 0.81–0.91（最高 0.9145）；組合多層快取後綜合命中率 71%、Tier 1 延遲 < 5ms、Tier 2 ~100ms。最反直覺的點：他原本擔心 entity swap 會讓 embedding 把不同資料庫的問題混為一談，餵錯答案——實測發現現代 embedding 模型把「Dragonfly 支援 WAIT 嗎」跟「Valkey 支援 WAIT 嗎」打到 0.63 分，根本不會誤判。

真正的失敗模式不是「沉默損毀」，是「過度保守」。如果你照業界慣例設 0.95 閾值，paraphrase 群裡有一段 0.80–0.92 的合法相似句被擋在門外——這些 query 該命中卻沒命中，每一筆都是一次 expensive cache miss。文章給了五個解法：降閾值（最直接但要量化風險）、namespace partitioning（用 metadata 分割再比對）、query rewriting（先 normalize 再 embed）、ensemble matching（多個 embedding 模型投票）、LLM-as-judge（borderline 那段 0.80–0.92 用 LLM 二次驗證）。設計重點從「絕對命中率」搬到「near-miss 分布」——你要看的不是有多少命中，是有多少差一點命中卻被擋下來。

以前討論 RAG 快取，主流恐慌是「embedding 太相似會讓系統靜默回錯」；於是大家把閾值設高、把命中率犧牲掉，誤以為這是安全。現在這份 28 對的實證資料把恐慌轉向：embedding 模型在 entity disambiguation 上其實夠強，真正在燒錢的是被高閾值擋下的合法 paraphrase。對 Henry 這種正在學 SQL 又在做 AI workflow 的人，這個觀察可以直接套到「Nexus 設計 token 查詢快取」這類場景——不是怕快取吐錯 token、是怕同一個 token 被三種寫法問三次卻一次都沒命中。

## AI Harness：把整個 codebase 當成 AI agent 的「上下文基礎設施」

你可以照 Activepieces 的開源實踐，把任何 monorepo 升級成「AI Harness」——七個元件：always-on instructions（~55 行的架構規則 + gotcha）、safety reflexes（每個 3–5 行的安全反射，像多租戶資料隔離、SSRF 防護）、on-demand context（每個模組 ~60 行的 feature doc，他們維護 40+ 份）、codified workflows（30–65 行的 slash command，例如 `/add-entity`、`/db-migration`）、scoped subagents（限縮工具權限的專職 agent）、external integrations（MCP 接 Linear / PostgreSQL / browser）、session hygiene（worktree 並行、context 清理）。實測數據：routine feature 的「第一次就對」成功率從 ~20% 升到 ~70%，校正循環減少 5 倍，目標是每個工程師每天 1 個 production-ready feature。

「AI harness」這個詞在挑戰一個直覺：AI agent 失敗的原因，多半不是模型不夠強，是它對你的 codebase 不夠熟。Activepieces 的拆法把「不夠熟」這件事拆成七種可文檔化的子題——架構規則寫成 `AGENTS.md`、團隊踩過的雷寫成 safety reflex、領域知識寫成 feature doc、複雜流程寫成 slash command、權限分割寫成 subagent。每一份都很短（55 行、3 行、60 行、30 行），但加起來就是一張完整的「我們公司怎麼寫程式」的說明書。當下一代模型發布，這份說明書直接吃滿新模型的能力，不必重寫。

以前要讓 AI 在你的 codebase 寫對程式碼，方向不外乎兩種：寫一份巨型 CLAUDE.md（什麼都丟進去、結果模型挑不到重點）或全靠模型自己讀檔（每次都要重新理解架構、token 一直燒）。AI Harness 的設計是把這兩極端轉成模組化的小檔——always-on 載入規則、on-demand 載入領域、slash command 載入流程——三層分工讓 context window 永遠裝著「該裝的」，不裝「以防萬一的」。對 Henry 這種已經在 Nexus monorepo 寫過 CLAUDE.md、自己 MCP server、自己 CLI 的人，這就是把零散的設定整合成一套有 leverage 的系統設計範式。

## 今日總結

今天的五則訊號全在問同一件事：**模型已經夠強，瓶頸在「上下文怎麼餵」**。

- **CC v2.1.126** 把模型清單交給 gateway、把 OAuth 流程降到「貼一行 code」——基礎設施摩擦歸零，CLI 才有存在感。
- **NgRx Signals Skill 評測** 把 skill 質量從「感覺對」變成「+16pp 通過率、$0.04 成本」——skill 是有 SLA 的工程產物。
- **CLI vs MCP** 揭示真實使用形態：跨 agent 平台時，portability 比結構化更值錢——先 CLI 後 MCP 是務實順序。
- **Semantic Cache 28 對實測** 把 RAG 恐慌轉向：embedding 沒在靜默損毀，0.80–0.92 那段才是真正的成本黑洞。
- **AI Harness 7 元件** 把「AI agent 在你 codebase 表現好不好」拆成可文檔化的七種子題——20% → 70% first-try success 是工程設計的成果，不是模型的功勞。

把這五個訊號合起來看：**AI Workflow 的成熟度，不在於工具多酷，而在於上下文基礎設施有多薄、多模組、多可量測**。Nexus 下一階段該做的不是「再加一個工具」，而是把已經有的 CLAUDE.md / MCP / CLI 重新拆成 always-on / on-demand / slash command 三層，並為每層加上 capability/trigger/cost 三軸的評測——讓「給 AI 看的 context」變成跟設計 token 一樣的可維護資產。
