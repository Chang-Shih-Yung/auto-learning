# 2026-03-28 學習日誌

## 1. jai — 讓 AI Agent 的爆炸半徑縮到工作目錄

Stanford 工程師 David Mazières 釋出 `jai`（409 pts，HN 今日 #1）——一個 12MB 的輕量 binary，讓 AI agent 命令在 filesystem isolation 下執行：工作目錄保持讀寫、家目錄改成 copy-on-write overlay、其他路徑全部唯讀。

這件事的起點是三個真實案例：Claude Code 刪掉 15 年家族相片、清空整個開發專案、Cursor 吃掉 100GB 的本機檔案。作者的論點是：「完整容器太重，完全放任太危險，jai 填補中間的落差。」原本一個失控的 agent 指令就能毀掉整個家目錄，現在只要在命令前加一個前綴——`jai claude-code`——失誤就只影響當前工作目錄，其他位置不動。

因為你已在跑 Claude_auto 排程 session 並讓 agent 自動寫檔、commit、push，jai 讓你可以對高風險操作（批次寫檔、git 重置）套上一層 filesystem 防護，不需要改變工作流，只是多一個前綴。

## 2. Claude Code v2.1.86（2026-03-27 釋出）

這次更新最直接影響 long-running session 的兩個修復：

- **`--resume` 失敗修復**：先前版本 `--resume` 偶爾無法接續上一個 session，現已修正。Claude_auto 排程 session 的連續性直接受益。
- **`/feedback` OOM 修復**：長時間 session 呼叫 `/feedback` 會觸發 out-of-memory crash，現已解決。你的 daily session 在結尾的 self-optimization 階段不再有崩潰風險。

其他修復：session ID header（proxy 聚合用）、Jujutsu / Sapling VCS 目錄排除、Windows config 不必要寫入（防 corruption）、OAuth URL 複製失效、窄終端機 masked input 顯示異常、macOS/Linux plugin 權限問題、多 instance 狀態列一致性。

原本你的排程 session 可能在最後一步（`/feedback` 觸發的 self-optimization）崩潰，導致 memory 更新丟失 → 這次修復讓 pipeline 從頭到尾能穩定跑完。

## 3. 3-Agent Pipeline：GitHub 活動 → Gemini → Notion + DEV.to（dev.to，45 reactions）

Yash Saini 用 [Mastra](https://mastraai.com/) 框架建了一個自動寫 blog 的 3-agent 系統：

| Agent | 工具 | Token 成本 |
|-------|------|-----------|
| Harvest | GitHub GraphQL API | 0（確定性函數） |
| Narrator | Google Gemini | 有（只在這裡用 LLM） |
| Publisher | Notion MCP + DEV.to API | 0（確定性函數） |

核心設計原則：**「只在必要時使用 LLM」**。原本：每週手動寫 blog 要花 1-2 小時，且常被遺忘 → 現在：GitHub Actions 每週日 08:00 UTC 自動觸發，產出含「週覽表」（commits / PRs / 程式碼行數）+ 活躍 repo 表格 + 敘事 blog 的 Notion 頁面，並同步推到 DEV.to，全程零人工介入。

因為你已建了 Claude_auto 的 fetch → process → write → commit 三段式 pipeline，這個架構是同一個 pattern——差別在 Mastra 給每個 agent 明確的 role 邊界，失敗時只需重跑失敗的那個 agent，而不是整個 session。

## 4. AI 軟體開發的未來：2 人決策 + Agent 執行（HN 討論）

Martin Fowler / Thoughtworks 研討會衍生的 HN 討論（[原連結](https://news.ycombinator.com/item?id=47062534)）湧現出幾個有具體論點的觀點：

- **simonw**：「代碼變成商品之後，真正有價值的是知道要建什麼、何時建的直覺。」
- **empath75**：新的團隊結構模型：「2 人做技術決策，由 agent 執行工作。」這不是比喻——是具體的人員編制建議。
- **crystal_revenge**：「AI 應該替代對軟體的需求，而不是替代工程師。」

一個有數字支撐的觀察：AI 在綠地專案的前幾週超高效，但隨著 codebase 複雜度線性增長，生產力曲線快速下降。沒有架構能力的人，AI 讓技術債累積速度比以前快，不是慢。

因為你的核心定位是「翻譯的角色」——把需求翻成規格、把設計翻成工程系統——這個模型正在說翻譯能力是護城河：懂得要建什麼，比懂得怎麼建更難被 agent 取代。

## 5. Next.js v16.2.1（2026-03-20 釋出）

這是一個 bug-fix backport 版，不含 canary 新功能。值得注意的修復：

- **Turbopack 三項修復**：webpack loader runner layer 崩潰修復、lazy require metadata + TLA（Top-Level Await）支援修復、worker_threads `eval:true` 模式修復。原本在這些邊緣案例下 dev server 可能靜默失敗或崩潰，現在穩定。
- **server actions standalone mode**：`cacheComponents` 設定下 server actions 失效的 bug 修復。
- **dynamic metadata routes**：adapter 輸出格式修復。

原本：dynamic import 或 TLA 的邊緣用法可能讓 `next dev` 靜默崩潰，且難以定位原因 → 更新後：Turbopack dev server 的這三個邊緣路徑被明確修復，local dev 更穩定。

因為你的 Claude_auto 專案用 Next.js App Router，Turbopack 的穩定性修復直接影響日常 `next dev` 的體驗。
