# 2026-03-28 學習日誌

## 1. 只把 LLM 用在真正需要它的地方——3-agent pipeline 設計模式

一個工程師在 dev.to 發文（52 reactions，今日 AI tag 第一），分享他如何用 3 個 agent 把 GitHub 每週活動自動轉成 Notion 工作日誌 + dev.to 文章，每週日 08:00 UTC 自動觸發，完全不需要手動回顧。

這個 pipeline 有一個讓人眼睛一亮的設計決策：三個 agent 裡只有一個用到 LLM。Harvest agent 用 GraphQL 確定性地把 commits、PR、code review、語言統計全部拉下來——不用 LLM，不會幻覺，不燒 token。Publisher agent 同樣是純函數，把結果平行寫進 Notion 和 dev.to。只有中間的 Narrator agent 才交給 Gemini，把 JSON 數據轉成第一人稱文章，還能切換 4 種語氣（casual、professional、technical、storytelling）。這個選擇讓每次執行省下 20–40 秒，而且 harvest 和 publish 兩個步驟的結果永遠可以預測。

以前「你上週做了什麼？」是週一站會前你要自己從記憶中拼湊的問題。現在這個問題在你起床前已經被 agent 回答完畢，工作記錄存進 Notion、文章發上 dev.to，格式乾淨、stats 準確，你不需要多花一分鐘。

## 2. Pieces for Developers——開發者工具的長期記憶層

Pieces 在 ProductHunt 上的定義是「the first AI-powered Long-Term Memory for developers」。用起來是這樣：你在 VS Code 裡查到一個解法、在 Chrome 裡複製了一段 API 文件、在 terminal 裡跑了一個你忘了參數的指令——它全部記下來，保留 9 個月以上，存在你自己的機器上，用自然語言搜尋。

機制是本地 AI 索引：Pieces 在背景持續監聽 IDE、瀏覽器、terminal、桌面應用的工作流，把每個片段做語意向量化，建立 deep link 讓你能直接跳回當時的原始脈絡。所有運算跑在你的設備上，不上傳任何資料，所以可以放心記錄內部專案的程式碼片段。

以前你花在找「那個 3 個月前解過的 TypeScript 泛型問題」的時間，平均比寫新解法的時間還長。現在一句 "TypeScript generic constraint 那個 error" 就能定位到當時的 Claude 對話、Stack Overflow 頁面、和你最後採用的那段 code，全部在 1 秒內出現，而且沒有任何一個 byte 離開你的電腦。

## 今日總結

今天兩篇文章指向同一個核心觀念：**把記憶和自動化的責任交給正確的工具，讓人和 LLM 都只做最適合自己的事。**

DevNotion 的教訓是架構上的：LLM 只用在需要語言理解的地方，其他步驟保持確定性。這個原則直接可以套用在 Claude_auto 的改版上——Step 1 和 Step 4 不需要 Claude，只有 Step 2 才值得燒 token。

Pieces 的教訓是工具選擇上的：開發者的工作記憶是一個真實的痛點，本地 AI 索引是解法，不需要上傳資料。你現在用 memory/ 檔案手工解決的問題，Pieces 做的是自動化版本。

兩者合在一起的方向：把 Claude_auto 的 Harvest 和 Publisher 做成確定性函數，同時讓 memory 層從手工 YAML 進化到有語意搜尋能力的索引。
