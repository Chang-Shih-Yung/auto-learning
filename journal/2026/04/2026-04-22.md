## Claude Code v2.1.117 — 原生 Binary 內建搜尋工具，subagent 分叉正式開放

Claude Code 今天更新到 v2.1.117。在 macOS 和 Linux 的 native build 中，Bash 工具現在直接內建 **bfs**（廣度優先搜尋）和 **ugrep**（Unicode-aware grep），不再需要系統環境裡裝有 ripgrep 或 find。以前每次 file search 都要 spawn 一個外部 process，現在是零額外 round-trip：在連續多步驟的 agent 任務中，這個差異從「感覺不到」變成「每 10 次搜尋就省下一次 process 啟動延遲的累積量」。此外，模型選擇現在可以跨重啟持久化——你選了 Opus 4.7，重啟後不再跳回預設模型。想測試 subagent 分叉的話，設定環境變數 `CLAUDE_CODE_FORK_SUBAGENT=1` 即可在外部 build 上啟用。

bfs 和 ugrep 以 native binary 形式嵌入 Claude Code 可執行檔，讓 Bash 工具直接呼叫而不依賴系統路徑。這個設計讓 Claude Code 在沒有安裝任何額外工具的環境（最小化 Docker 容器、CI runner、新開的 MacBook）也能保持一致的搜尋效能。同時，plugin 安裝流程加入了自動解析遺失依賴的機制，並在 install / update 階段就強制執行 marketplace 白名單驗證——不合規的 plugin 在安裝時就被擋住，而不是在執行時才崩潰。

以前，Claude Code 的 Bash 搜尋效能取決於宿主環境：CI 上沒裝 ripgrep 就慢，換一台新機器就得重新設定。現在，搜尋能力跟著 binary 走，**環境一致性是預設值，不是需要維護的狀態**。模型選擇持久化則消除了另一個隱形摩擦：每次重啟都要重新選模型，現在不用了。subagent 分叉的開放意味著 Claude Code 在 external build 上也能正式把複雜任務拆成並行的子代理——這是向 multi-agent 工作流邁出的一步。

## Vercel 資安全鏈解析：一個 Roblox 外掛 → 一個 AI 工具 → $2M 勒索

如果你的 Vercel 專案曾授予任何第三方 AI 工具「Allow All」的 OAuth 權限，現在就去 Vercel Dashboard → Settings → Integrations 逐一審查。這次事件的受害者是 Vercel 本身：攻擊者最終拿到的是 Vercel **未被標記為 sensitive 的 environment variables**，向 Vercel 索取 $2M 贖金。整條攻擊鏈從一個 Roblox 遊戲外掛開始，Vercel 是最後一個倒下的骨牌。

整條攻擊鏈起點不是 Vercel，而是 Context.ai（一個 AI 辦公室套件）的一名員工——他在 2026 年 2 月下載 Roblox 遊戲作弊程式時，順帶安裝了 **Lumma Stealer** 惡意軟體。Lumma Stealer 竊取了該員工的瀏覽器 session token 和 OAuth refresh token。攻擊者接著用這些憑證存取 Context.ai 的服務帳號，再利用 Context.ai 對某名 Vercel 員工 Google Workspace 的「Allow All」OAuth 授權，橫向移動進入 Vercel 的內部環境，取走環境變數。漏洞的根源不是 Vercel 的程式碼，而是一個被遺忘的「Allow All」授權按鈕。

以前，第三方 AI 工具的 OAuth 授權像一個被忽略的後門——申請時按了「Allow All」，之後就忘了它存在。這次事件明確了一件事：**你授予給 AI 工具的權限，等於授予給它的整個員工網路和供應鏈**。Before：OAuth 是一次性設定。After：OAuth 是持續需要審計的風險面。如果你的 Claude_auto 或 Nexus 在 Vercel 上開著 AI 工具的 All-permission 授權，週末就是做最小權限審查的好時機。

## manifest — 宣告式模型路由，Personal AI Agent 成本削減最高 70%

如果你在用 Claude Code 或自建 AI agent 工作流，manifest 讓你用一份 YAML 設定智慧路由規則：簡單的程式碼補全自動走 Haiku（比 Opus 便宜約 50 倍），需要複雜推理的任務才送 Opus 4.7。官方宣稱在混合任務場景下成本削減最高 **70%**。manifest 目前有 **5,483 stars**，今天單日新增 177 stars——對一個近期剛出現的工具來說，這個速度代表社群正在快速驗證它的實用性。

manifest 的核心是一個 routing layer：你用 YAML 宣告任務的複雜度閾值（例如：「上下文長度 < 2,000 tokens 且無 function call → 路由到 Haiku；否則 → Opus 4.7」），manifest 在每次 API 請求前評估這些條件，再選擇對應模型。它不是用另一個 LLM 來做路由決策（那樣會產生額外 token 費用），而是讓開發者**宣告式地定義規則**——這讓路由邏輯透明可 debug，也可以 commit 進版本控制，在 code review 中討論成本策略。

以前，個人開發者建 AI agent 面臨一個兩難：用好模型太貴，用便宜模型又怕品質掉，通常是在專案初期選一個模型然後全部套用。manifest 把這個選擇從「一次決定、所有任務共用」變成「任務層級的動態決策」。Before：你選一個模型，成本結構固定。After：每個任務用剛好夠用的算力，帳單自動最佳化，帳單結構跟任務複雜度成正比，而不是跟任務數量成正比。

## Ternary Bonsai — 1.75GB 跑 8B LLM，M4 Pro 上 82 tok/s

Ternary Bonsai 的 8B 模型在 M4 Pro 上只需要 **1.75GB RAM**，推理速度 **82 tokens/sec**。相比之下，同等級的標準 16-bit 8B 模型需要 16GB RAM、速度慢 5 倍。如果你的 MacBook 是 16GB 機型，以前跑 8B 模型幾乎佔滿半個 unified memory；現在 Ternary Bonsai 讓這個記憶體成本縮到 **11%**，剩下的空間繼續跑 IDE、Docker container 和你的其他工具。benchmark 平均分 75.5，僅輸給使用全精度的 Qwen3 8B（但 Qwen3 8B 需要 16.38GB）。

「Ternary」指的是模型權重只有三個值：{-1, 0, +1}。標準 16-bit 浮點模型的每個權重需要 16 bits 儲存；ternary 模型理論上每個權重只需要 1.58 bits（log₂3）——**9 倍的記憶體壓縮比**。Ternary Bonsai 在全網路使用 ternary 權重（不是只量化部分層），並透過訓練技術補償精度損失。這個設計也讓 iPhone 17 Pro Max 可以跑同一個 8B 模型——速度 27 tok/s，能耗 0.132 mWh/tok，比 16-bit 模型省 3–4 倍電。

以前，在 MacBook 上跑「夠用的」本地 LLM 是一個取捨遊戲：品質好 → 記憶體爆表，記憶體夠用 → 只能跑 3B 小模型。Ternary Bonsai 打破了這個取捨：8B 的推理能力、接近 3B 的記憶體佔用。Before：本地 AI 需要高記憶體設備才能跑有用的模型。After：同樣的 16GB MacBook，可以同時跑 8B LLM + 完整開發環境，資料不出機器，風扇不狂轉，帳單不進 API 計費。

## 今日總結

今天的四篇文章在表面上看是四個不同話題，但底層有一條共同的設計原則：**把成本和複雜度推到最合適的位置**。

- Claude Code v2.1.117 把搜尋工具嵌入 binary，把環境依賴的複雜度內化到工具本身，讓上游使用者感受不到它
- manifest 把模型選擇成本推到路由規則層，讓每個任務用剛好夠用的算力
- Ternary Bonsai 把推理成本推到 ternary 量化層，讓記憶體佔用縮到最小點
- Vercel 資安事件則是反例——把信任成本留在一個被遺忘的「Allow All」按鈕上，最終代價是 $2M

這四個故事合在一起，是一個關於「成本歸屬」的完整思考框架。應用點：下次在設計 Nexus MCP Server 的工具呼叫策略時，問自己：這個複雜度放在哪一層最合適？這個授權的最小必要 scope 是什麼？
