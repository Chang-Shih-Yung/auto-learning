## Cloudflare Code Mode：把 AI 與 API 的對話從 117 萬 token 壓到 1,000

你可以讓 Claude 同時操作 Cloudflare 的 2,500 個 API endpoint，整個過程消耗的 token 永遠是固定的 1,000——不管你用到哪個端點、做幾個組合操作。如果你在 Nexus 裡已經建了 MCP server，現在可以把它接上 Cloudflare Code Mode，讓 Claude 在一次工具呼叫裡完成十個 API 操作，而不是十次分開呼叫、十次 context 消耗。

傳統 MCP 的做法是把所有 API endpoint 的 schema 整包載入 context——2,500 個端點就直接吃掉 1.17 百萬 tokens，context window 在開始工作之前就已經被工具清單塞爆。Code Mode 的解法是把整套 API 暴露成僅僅兩個工具：`search()` 和 `execute()`，Agent 不是查工具清單，而是直接寫 TypeScript 程式碼，讓 SDK 在執行時才實際呼叫對應的 endpoint。查 API 文件的工作從 context 搬到了 execution time，token 成本永遠固定在 1,000。

以前你用 MCP 連 API，需要提前把所有工具定義塞進 context，工具越多 token 越貴，最後只能取捨哪些端點值得連、哪些暫時省略。現在，不管你連 10 個還是 2,500 個端點，context 成本固定是 1,000 tokens——你的 Agent 可以操作任何一個端點，卻完全感覺不到 context window 的壓力。這是 MCP 從「工具目錄」轉向「程式碼代理」的關鍵轉折點。

## Google Cloud Next 2026：Vertex AI 退場，Gemini Enterprise Agent Platform 登場

你可以用 Google ADK v1.0 的 TypeScript SDK 建立 graph-based multi-agent 系統，把 Claude、Gemini 和你自己的業務邏輯放在同一張圖裡，讓各個 sub-agent 分工執行，然後部署到任何 container 環境，不被鎖定在單一雲端平台。在 Cloud Next 2026（4 月 25–27 日）宣布的 Workspace Studio，則讓非工程師也能用自然語言描述「每週五提醒我更新追蹤表」，Gemini 自動在 Gmail / Sheets / Drive 建好自動化流程。

Google 把 Vertex AI 改名為 Gemini Enterprise Agent Platform，把 Agentspace 也並入同一套產品。ADK v1.0（Python / Go / Java / TypeScript 都已正式穩定）是 code-first 框架：你定義 agent graph 的節點、sub-agent 的工具集、節點之間的路由邏輯，框架負責把請求派到正確的模型——200 多個模型都在目錄裡，包含 Anthropic Claude。Memory Bank 讓 agent 的記憶跨 session 持久化，能撐好幾天的長任務，不依賴 context window 的短暫存在。

以前要建 multi-agent 系統，你要自己寫 orchestration 邏輯、自己處理模型路由、自己實作 session 持久化——每個部分都要從頭設計，而且換模型就要改程式碼。現在 ADK v1.0 把這些變成框架功能，TypeScript SDK 讓你用最熟悉的語言介接，同一套 agent graph 可以路由到 Claude 或 Gemini，不用被綁死在單一廠商。Vertex AI 的消失，代表 Google 下的賭注是 agent-native 介面，不是模型租用介面。

## GenericAgent：AI Agent 也可以有技能樹，30K token 對決競品的 200K

你可以用 GenericAgent 讓你的 AI agent 每次完成任務後，自動把成功的執行路徑「晶化」成技能存進個人技能樹——下次遇到類似任務，直接呼叫 SOP，不用從頭推理。整個框架只有 3,000 行程式碼，context 永遠壓在 30,000 tokens 以下，而競品框架（LangChain、CrewAI 等）動輒需要 200,000 到 100 萬 tokens 才能做同樣的事。GitHub 目前 6,100 stars，arXiv 論文同步發布（2604.17091），4 月剛加入 cron scheduler 整合與 session archive memory。

GenericAgent 的核心概念是「情境資訊密度最大化（Contextual Information Density Maximization）」：layered memory 架構讓 agent 預設只看小的 high-level 索引，需要細節才往下鑽——Meta Rules → Insight Index → Global Facts → Task SOPs → Session Archives，五層按需載入，每層只在必要時才進入 context。9 個 atomic tool 加上約 100 行的 Agent Loop 處理所有事，從瀏覽器、terminal 到 ADB 設備控制都在這個骨架裡。每次任務結束，驗證過的執行路徑自動壓縮成可重用的程式碼技能——不只存在記憶裡，而是變成下次可直接執行的工具。

以前你的 AI agent 每次都從零開始：相同的任務要重新推理、相同的工作流要重新組裝、context window 被歷史紀錄吃掉大半，然後繼續燒 token。現在 skill tree 讓 agent 愈用愈快——第 100 次執行相同任務的 token 消耗，比第 1 次少 6 倍，而且這棵技能樹完全屬於你，不在雲端、不綁訂閱、不會因為 API 改版而重置。Claude Code 的 /skills 系統走的是相同路線，只是 GenericAgent 把這個設計原則做成了獨立的 open source 框架，任何 LLM 都可以用。

## 今日總結

今天三個主題都在解同一道題：**token 是稀缺資源，怎麼設計「不用 token 的方式」才是真正的競爭力**。

- **Cloudflare Code Mode** 的解法是重新設計**介面**：把 2,500 個工具壓成 2 個，讓 agent 寫程式取代查清單，token 成本從 117 萬壓到 1,000、固定不變。
- **Google ADK v1.0** 的解法是重新設計**框架**：把 orchestration 變成宣告式 graph，讓模型路由成為配置而非程式碼，Memory Bank 把持久化從 context window 搬到資料庫層。
- **GenericAgent** 的解法是重新設計**記憶**：讓每次成功執行自動晶化成技能，下次直接 apply，不用重跑推理——30K 對決 200K，6 倍的差距來自設計，不來自硬體。

共同前提：**context window 不是用來存歷史的，是用來做決策的**。三個工具，三種方式，把本來塞在 context 裡的東西搬到它應該在的地方。
