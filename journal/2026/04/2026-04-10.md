## Claude Code v2.1.98：安全加固 + MCP 結果限制大幅提升

你用 Claude Code 跑 bash 命令的時候，現在多了一層 Linux 子程序沙盒保護——v2.1.98 引入 PID namespace 隔離，把每個 bash subprocess 關進獨立的命名空間，讓惡意命令或意外的 fork 無法觸及外部進程。同一版本也修掉了兩個嚴重的安全漏洞：一是用 backslash 跳脫 flag 可以把危險命令偽裝成「唯讀」被自動放行；二是複合命令（`cmd1 && cmd2`）可以繞過權限檢查。如果你的 MCP server 在傳大型 schema 或 DB 結構，現在可以透過 `_meta["anthropic/maxResultSizeChars"]` 把結果上限拉高到 **500,000 字元**，終結截斷問題。

這兩個漏洞的修復說明一件事：prompt-injection via shell 一直是 AI coding 工具最危險的攻擊面。`backslash + flag` 這種技巧會讓 Claude 把 `rm -rf /` 當成唯讀操作放行——安全設計不能只靠 heuristic，必須在 subprocess 層做真正的隔離。PID namespace 是 Linux container 的基礎技術，跟 Docker 的 sandbox 邏輯相同，差別是這次直接整合進 CLI 而不需要額外的 container 環境。

以前：bash 命令逃脫 Claude Code 的 permission 審查只需要一個反斜線，MCP 傳大型 schema 必須自己切片、自己組合。現在：subprocess 級別的 namespace 隔離讓權限繞過從「容易」變成「需要 kernel 層漏洞」；而 Nexus 元件庫的完整 token schema 一次傳完，不再截斷。

## Research-Driven Agents：先讀論文再動手，$29 換來 15% 加速

你可以花 **$29**（$20 VM + $9 API）在 3 小時內讓 AI agent 自主找出 5 個真正有效的推論優化——前提是先讓 agent 讀文獻和競品程式碼，再讓它動手改。SkyPilot 的研究團隊做了 30+ 個實驗，最終在 x86 Intel Xeon 上把 TinyLlama 1.1B 的文字生成速度提升 **15.1%**（41.37 → 47.62 tokens/s），ARM Graviton3 提升 **5%**（94.07 → 98.77 tokens/s）。所有成功的優化都指向記憶體效率，而非 compute：Softmax fusion 把 3 次 pass 壓到 1 次、RMS norm fusion 把 2 次壓到 1 次、Flash attention KQ tile fusion 減少 memory 帶寬壓力。

純 code-only 的 agent 嘗試的是 SIMD prefetching 和 loop unrolling——這些都在統計誤差範圍內，沒有實際效果。加入文獻引導之後，agent 開始問的是「能不能把這幾個操作合併成一個 memory pass？」——這種問法只有讀過 CUDA/Metal 優化論文的人才會想到。更有趣的發現是：**研究 fork 和競品 backend 比翻 arxiv 更有生產力**，五個成功優化中有兩個直接來自競品程式碼的模式，而不是學術論文。

以前：要 AI agent 優化底層推論，你就 prompt 它叫它想辦法，它猜幾個常見技巧，全都在誤差範圍內，沒用。現在：先給 agent 一個「研究 + 閱讀」階段，讓它主動讀 LLVM.cpp、Metal shader、競品 inference 實作，它才能問出正確的問題，然後真的找到有效的優化路徑。

## 今日總結

今天兩篇文章都在同一個方向上交匯：**AI agent 的安全與可靠性**。

Claude Code v2.1.98 的安全修補提醒我們，AI coding 工具的最大風險不是 AI 寫錯程式，而是 AI 在 shell 層被繞過——而解法是真正的 OS 隔離（PID namespace），不是 heuristic 過濾。

Research-Driven Agents 的研究則示範了另一種可靠性：讓 agent 在動手前先「讀」，不只是讀你給它的 prompt，而是主動研究外部文件和競品實作。這個模式和 learning-context.md 中「先 Read sources.yaml 再執行」的設計邏輯完全相同——你已經在實踐 Research-Driven Agent 架構，只是沒有這個名字。

**應用點**：下次設計 Claude Code hook 或 MCP tool 時，可以思考兩件事：(1) 這個 tool 的 subprocess 有沒有隔離？(2) 執行前有沒有「research phase」讓 agent 讀相關上下文再動手？
