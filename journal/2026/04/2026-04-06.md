# 2026-04-06 學習日誌

## 1. Claude Code v2.1.92：Write tool 快了 60%，/cost 現在拆得開

你的 Claude Code 寫入大型檔案的速度比上個版本快了整整 60%——一個 2000 行的 TypeScript 檔案，過去拖拖拉拉，現在幾乎是瞬間完成。同版本還加入了 `/cost` 指令的 per-model 分拆：訂閱用戶可以看到每次對話裡每個模型（Sonnet、Haiku、Opus）各花了多少 token，以及 cache hit 佔比，不再是一個模糊的總費用數字。Interactive Bedrock setup wizard 讓 AWS 驗證流程從手動填 config 變成問答式引導，適合同時要對接企業 AWS 環境的開發情境。

這次 Write tool 的效能改善是內部 I/O 路徑的重構，原本每次寫入都有多餘的 buffer 複製步驟。Per-model cost breakdown 的背後是 Claude Code 將每次 API 呼叫的 response metadata 做了精細記錄，再聚合到 `/cost` 指令輸出——cache hit 明細讓你知道 prompt cache 是否真的在節省你的錢。

以前你看 `/cost` 只知道「這個 session 花了多少」，不知道哪個模型最貴、cache 有沒有命中。現在你拿到的是完整的費用帳單，可以調整哪些 prompt 值得快取、哪些 subagent 換小模型跑就夠了。Write tool 快 60% 這件事在排程任務場景感受最明顯——batch 寫多個檔案的任務，總時間壓縮比例接近這個數字。

## 2. ctxlint：你的 CLAUDE.md 有 74% 可能是廢話

你可以用 `ctxlint` 在 100ms 以內掃描你的 CLAUDE.md（或 AGENTS.md），找出哪些行在浪費 AI agent 的 token 預算。在 8 個真實開源 repo 上的測試顯示，平均有 74% 的 context file 內容是可以刪掉的——stale 的檔案路徑、agent 自己 `ls` 就能找到的目錄樹、config 檔裡已有的技術棧描述、README 裡重複的段落。工具的精準度是 91%，跑完後給你一份 signal-to-noise ratio 報告，標注每一個問題類型。真實案例顯示最多可以削減 67% 的 token 成本。

ctxlint 完全不用 AI model 做分析。它靠確定性邏輯：用 filesystem 檢查找 stale 路徑、用 JSON parsing 對比 package.json、用行數計算偵測 excessive length。這讓它在 10,000+ 個檔案的 repo 上也能在 100ms 內完成，不帶額外的 API 費用。8 種問題類型（stale references、stale commands、directory trees、inferable stack、redundant README、style guide rules、excessive length、token budget）覆蓋了 context file 最常見的膨脹原因。

以前你寫 CLAUDE.md 靠直覺，不知道哪些描述 AI 其實根本不看。現在你有一個可以跑進 CI 的 linter，在每次更新 CLAUDE.md 後自動告訴你有沒有引入新的 token 浪費——和你在前端工程用 ESLint 檢查 TypeScript 的邏輯完全一樣，只是目標從程式碼換成了 AI 的工作說明書。

## 3. git worktrees 並行 Claude Code：把 20 分鐘壓縮成 10 分鐘

如果你的 Claude Code 計畫裡有 6 個彼此獨立的 batches，你可以用 git worktrees 把它們並行跑——每個 worktree 是同一個 repo 的獨立工作目錄，各自 checkout 到不同的 branch，互不衝突。一個 20 分鐘的計畫，6 個 batch 全部同時跑，實際 wall-clock 時間壓縮到約 10 分鐘。作者把這個模式包成了 `cast-parallel` 工具，加入了 signal trap 防止孤兒 process、merge conflict 保留而不自動解決。

git worktrees 是 git 原生功能，讓同一個 repo 可以有多個 working directory 同時存在、各自指向不同的 branch。Claude Code 在每個 worktree 裡獨立運作，看到的是完整的 repo 歷史但操作的是各自的 checkout——不會有兩個 session 同時寫到同一個檔案的競爭問題。完成後再用 `git merge` 或 cherry-pick 把各 branch 的結果合回主線。

以前你用 Claude Code 跑複雜計畫，只能等 Batch 1 跑完才能開 Batch 2，時間是線性疊加的。現在你可以讓 Claude Code 的多個 session 像你在企業系統裡用 RxJS `forkJoin` 並行打 API 一樣——同時出發，等最慢的那個完成就好。API token 會翻倍，但對 6 個 batch 以上的計畫，時間收益遠大於 token 成本。

## 4. Ghost Pepper：按住 Control 說話，1-2 秒後自動貼上乾淨文字

你在 Mac 上按住 Control 鍵說話，放開後 1-2 秒，轉錄好的文字自動貼到游標所在位置——不需要網路、不上傳任何語音資料。Ghost Pepper 是 100% 本地運行的 hold-to-talk 語音輸入工具，語音識別用 WhisperKit，文字清理用 Qwen 0.8B（預設選項）。整個流程包含去除 "嗯"、"啊"、自我修正等口頭禪，輸出的是已整理過的乾淨文字，可以直接貼進程式碼編輯器或任何文字欄位。模型在第一次使用時下載到本地，之後離線可用。

Ghost Pepper 的架構分兩層：WhisperKit 處理 ASR（Automatic Speech Recognition），跑在 Apple Silicon 的 Neural Engine 上，tiny 模型只有 75MB；Qwen 0.8B 做 post-processing，把「所以說……我的意思是……這個 function 要……處理 null 的情況」清理成「這個 function 要處理 null 的情況」。兩個模型都是 LLM.swift 呼叫，完全在本機的 Metal 硬體加速，不用 Python 環境、不用 Docker。

以前你要在編輯器裡用語音輸入，只能靠 macOS 內建的 Dictation（需要網路、沒有 filler word 清理）或 OpenAI Whisper API（快，但每段語音都上傳到雲端）。現在你有一個 1-2 秒回應、資料完全不外洩的本地選項，在寫 CLAUDE.md、commit message、PR description 的時候可以直接說給它聽。

## 5. Freestyle：700ms 啟動的 Linux VM，專為 coding agents 設計的沙盒基礎設施

你可以透過 API 在 700ms 內為每個 coding agent 任務啟動一個完整的 Linux VM——擁有 root 存取、完整 Linux 網路堆疊、支援 Docker 和 KVM nested virtualization。Freestyle 是一個以 agent 工作流為核心設計的基礎設施平台，最特別的功能是 **live forking**：你可以在 VM 運行中的任意時間點，用毫秒級的速度 clone 出一個完全相同的狀態快照，不需要暫停執行。Idle 的 VM 可以 pause，暫停期間完全不收費。

Freestyle 的 live fork 技術類似作業系統的 copy-on-write 記憶體頁面機制，在記憶體與磁碟層做增量快照而不是完整複製。700ms 的啟動時間靠的是預熱的 VM image pool——不是從零 boot，而是從已初始化的快照恢復。對 coding agents 而言，這意味著每次「嘗試一種實作方式」的成本從分鐘級變成亞秒級，而且每個嘗試都在完全隔離的環境裡，不會污染彼此的狀態。

以前 coding agent 要在沙盒環境裡執行程式碼，要麼等 Docker container 啟動（10-30 秒），要麼在共享環境裡冒著互相干擾的風險。現在 Freestyle 讓 agent 在 700ms 內拿到一個真實的 Linux 環境，跑完了 pause 掉不計費，下次需要的時候從快照 fork——像給每個 AI 任務各自開一個乾淨的虛擬機，但速度和成本都像用容器一樣。

## 今日總結

今天五篇文章有三篇直接指向同一個核心問題：**如何讓 Claude Code 用得更有效率**。

- **v2.1.92** 給你費用透明度和更快的 Write tool——這兩個改善都直接影響你的排程任務效率。
- **ctxlint** 讓你把 CLAUDE.md 當程式碼維護，用 linter 而不是直覺來管理 AI 的工作說明書。91% 精準度、67% token 削減，這不是玩具，是可以進 CI 的工具。
- **git worktrees 並行**把你對 RxJS `forkJoin` 的既有理解直接平移到 Claude Code 工作流——你已經懂這個模式，只是需要換一個執行環境。

Ghost Pepper 和 Freestyle 是兩個不同方向的延伸：一個是讓你用語音輸入降低鍵盤摩擦，一個是讓 coding agent 拿到真正隔離的執行環境。後者對你的容器學習路徑是一個很好的 mental model——700ms VM 不是容器，但它解決的問題（隔離、快速啟動、無狀態）和你學 Docker 的理由是一樣的。
