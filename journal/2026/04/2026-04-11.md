## Claude Code v2.1.101：長 session 不再失憶，新增 `/team-onboarding` 指令

你的 Claude Code 長 session 現在不會在 resume 時丟失 context 了。v2.1.101 修復了大型對話 resume 時的 context loss bug、長 session 的 memory leak，同時加了 `/team-onboarding` 指令——一鍵產生新成員上手指南，輸出可直接 commit 進 repo。還有一個值得注意的安全修復：POSIX `which` fallback 存在 command injection 漏洞，這版已一起打掉。

這些 patch 背後的機制：context loss 是因為大型對話在 resume 時序列化的 snapshot 沒有正確重建；memory leak 是 long-running session 沒有及時釋放中間狀態的 reference。v2.1.101 同時修了 rate-limit 提示（現在會顯示是哪種 limit 被觸發），以及 Bedrock SigV4 認證失敗的問題——用 AWS Bedrock 跑 Claude 的 enterprise 用戶終於可以正常使用。

以前你開一個涉及大型 codebase 的長 session，resume 後 Claude 的記憶會出現空洞，得重新補充 context——有時等於重新開始。現在 context 在 resume 後完整保留，你的多小時 session 不再需要手動 checkpoint。`/team-onboarding` 則把以前需要手工維護的 onboarding wiki，變成一個指令就能更新的 markdown 產物。

## n8n：183K stars 的 TypeScript 工作流平台，現在原生支援 MCP

你可以用 n8n 把 Claude 接到 400+ 個服務——GitHub、Slack、Linear、資料庫、任何 REST API——不用寫一行 glue code，在瀏覽器裡拖拉節點就能完成自動化流程。最新版本把 MCP client 和 MCP server 都建進來了，代表 n8n 可以呼叫 Claude 的工具，也可以把自己的 workflow 暴露給 Claude 當工具用。全部可以 self-host，資料不離開你的伺服器。今天在 GitHub Trending 排名第七，單日新增 253 顆星。

技術基礎是 TypeScript，核心是 visual node graph，每個節點是一個 integration。MCP client 讓 n8n 在 workflow 裡直接呼叫 Claude Code 的工具集；MCP server 模式讓你把整條 n8n workflow 暴露成一個 Claude 可以呼叫的工具——等於把 400+ 個 integration 的能力，一次打包進 Claude 的工具清單。183,489 stars、56,663 forks，是目前 GitHub 上最熱的開源 workflow automation 平台。

以前要讓 Claude 自動發 GitHub PR、通知 Slack、更新 Linear ticket，你得分別寫三個 API 整合、處理 auth、處理 error retry。現在 n8n 把這些全部打包好，Claude 透過 MCP 直接呼叫，不需要寫 glue code，整條鏈在你的機器上跑，沒有資料外洩的疑慮。

## Linux Kernel 正式確立 AI 程式協助的貢獻格式：`Assisted-by` 標籤

你現在用 Claude Code 寫的程式，有了可被 Git 追蹤的標準格式。Linux kernel 本週正式發布 AI coding assistant 使用規範，規定所有 AI 協助的 contribution 必須加上 `Assisted-by: Claude:claude-3-opus` 標籤——這個格式正在成為開源社群的事實標準。HN 上有 204 分、141 則討論，是今天最熱的技術政策討論之一。

規範的核心規則有兩條。第一：AI agent 禁止自行加 `Signed-off-by`（那是人類對 Developer Certificate of Origin 的法律聲明）；所有 Signed-off-by 必須由人工 review 並負責。第二：必須加 `Assisted-by: AGENT_NAME:MODEL_VERSION [工具清單]`，工具清單可列 coccinelle、sparse、smatch 等分析工具，但排除 git、gcc 這類基本工具。完整範例：`Assisted-by: Claude:claude-3-opus coccinelle sparse`。讓 code review 者能追蹤哪些部分有 AI 參與。

以前 AI 生成的 code 混進 commit 沒有任何標記，review 者不知道哪些邏輯是 AI 寫的、哪些是人工驗證的——這讓 code review 的風險評估變得盲目。現在有了標準格式，你的 Nexus monorepo 裡「60% AI 生成」的 component，可以在 commit message 裡留下可追蹤的痕跡。往後 team PR review 時，reviewer 立刻知道哪段 code 要更仔細看，review 的重點不再憑感覺，而是有標籤為依據。

## 今日總結

今天三篇文章圍繞著同一個核心主題：**AI 工具正在被「制度化」**。

Claude Code v2.1.101 修的不只是 bug，而是讓長時間工作流變得可靠——context 不丟、memory 不漏，這是 AI 工具從「玩具」走向「基礎設施」的必要條件。n8n 的 MCP 原生整合，把 400+ 個服務的接入成本降到幾乎為零，讓 AI 工作流的「最後一哩路」有了標準化的解法。Linux kernel 的 `Assisted-by` 標籤，則是整個產業開始認真對待 AI attribution 的信號。

對 Henry 來說，這三件事的交叉點是：**你的 AI 工作流已經夠成熟，需要的不是更多新工具，而是讓現有工具變得可靠、可追蹤、可協作**。接下來值得投入的方向：把 n8n 的 MCP 和 Claude Code 的排程任務串起來，同時在 Nexus 的 CI/CD 裡加上 `Assisted-by` attribution，讓 60% AI 生成的 component 有了可審計的歷史記錄。
