## Anthropic 首次公開 Claude Code 品質退化事後分析

## Claude Code v2.1.119：設定持久化、GitLab MR 支援、MCP OAuth 修復

Claude Code v2.1.119 上線後，你在 TUI 裡用 `/config` 調的設定（theme、editor mode、verbose 等）會自動寫進 `~/.claude/settings.json`，下次重開 session 一樣生效；同時 `--from-pr` 現在支援 GitLab Merge Request URL，意味著你可以直接把 Nexus 的 MR 帶入 Claude Code 做 code review，不再只能用 GitHub。此外 MCP OAuth HTTP 連線失敗的邊界 bug 已修復，subagent MCP server 重新設定時改為並行連線，啟動更快。

設定持久化採用三層 override 優先序：`policy > local > project`，讓不同專案可以各自覆寫全域設定，企業 policy 又能統一鎖定關鍵行為。`--from-pr` 的多平台支援是改抓 git remote 的 host 而非寫死 github.com，所以 GitLab Enterprise、Bitbucket 都自然支援，不需要特殊 flag。

以前：每次開新 session，你的 Claude Code 設定都要重設一次，CLI 工具感覺不像自己的工具，像個陌生人的機器。現在：`~/.claude/settings.json` 讓 Claude Code 開始有「個人化記憶」——你的 theme、editor mode、偏好 model 都跟著你走，而不是跟著 session 消失。搭配 project 層的 `.claude/settings.json`，Nexus 和 Claude_auto 可以各自有不同設定，全局不打架。

你現在可以在 ChatGPT Plus/Pro 和 Codex 裡用 GPT-5.5。它在 Terminal-Bench 2.0 拿到 82.7%、SWE-Bench Pro 跑到 58.6%（一次 pass）、BrowseComp 拿到 90.1%，OSWorld-Verified（真實電腦操作）達到 78.7%。換成直覺語言：十個工程任務，它可以一次搞定差不多六個，不用你手動分解。HN 上這則消息拿到 1,217 pts，是今日最高分。

GPT-5.5 是完全重訓的模型，針對「多步驟跨工具任務」做強化——它能在寫程式、查資料、操作軟體之間切換，直到任務完成為止。OpenAI 的關鍵技術選擇是讓它用「更少 token 完成相同任務」，per-token 延遲跟 GPT-5.4 持平，但完成效率更高，這意味著在 API 計費上它理論上更便宜。NVIDIA 已在 Codex 基礎設施直接部署 GPT-5.5。

以前：你用 Claude Code 做 agentic coding，GPT 系列更像是逐步指導的助手，需要你主動拆任務。現在：GPT-5.5 進入同一個 agentic 競技場——Claude、GPT 都能跨工具自主跑任務，這對你的意義是：你選擇 AI 工具時有了真正的比較基準，而不再只能憑感覺說哪個比較「好用」。

## A2A Protocol v1.2：150 個組織在用，MCP 的跨平台拍檔

Google 在 Cloud Next 2026 宣布 A2A Protocol 升到 v1.2，現在有 150 個組織在正式環境使用（原本 50 個合作夥伴）。最直接的意義：你的 MCP Server 解決「agent 連工具」的問題，A2A 解決「agent 跟另一個 agent 跨平台溝通」的問題——兩個協定同時用，才能覆蓋完整的 multi-agent 場景。v1.2 新增了 cryptographic signature 的 agent card，讓跨組織的 agent 信任可以被驗證。

A2A 讓不同 framework 和不同廠商的 agent 能互相溝通——Salesforce 的 Agentforce agent 可以透過 A2A 把任務交給 Google Vertex AI 上的 agent，Vertex 再查 ServiceNow agent，三個系統不需要互相了解彼此的內部架構。A2A 已移交 Linux Foundation 的 Agentic AI Foundation 治理，和 Anthropic 的 MCP 一起放在同一個治理組織下（MCP 本身已達 9,700 萬次安裝）。

以前：你的 MCP Server 只能讓 Claude 呼叫你自己建的工具，agent 的邊界就是你的工具邊界。現在：如果 A2A 成為標準，你的 agent 可以呼叫任何一個支援 A2A 的外部 agent——組合出超過任何單一工具集的能力邊界，而且這件事不是未來計畫，是 150 個組織已在生產環境跑的現實。

## Bitwarden CLI 供應鏈攻擊：GitHub Actions 成為 npm 注毒入口

你需要確認你的 CI/CD 流程有沒有在 4/22 台灣時間下午 6–7:30（美東 5:57–7:30 PM）安裝過 `@bitwarden/cli@2026.4.0`——這個版本在那 90 分鐘內包含了惡意程式碼。受影響的機器，攻擊者會竊取：加密錢包私鑰、GitHub token，並利用這些 token 向你的 CI/CD pipeline 注入惡意 GitHub Actions workflow、把 secrets 傳到偽裝成 Checkmarx 的域名 `audit.checkmarx[.]cx`。

攻擊者入侵了 Bitwarden 的 CI/CD pipeline 中的一個 GitHub Action，讓它在 npm publish 階段把 `bw1.js` 換成惡意版本，再用 AES-256-GCM 加密竊取到的資料後外洩。核心漏洞：GitHub Action 的 GITHUB_TOKEN 本質上是 CI/CD 流程的 root 權限——一旦這個 token 被攔截，攻擊者就能讀取所有 repository secrets 並修改 workflow 定義。

以前：你認為 npm 上的官方套件是安全的，Bitwarden 是知名開源密碼管理器，下載就是用。現在：官方的 npm 套件也可以透過 CI/CD pipeline 被污染——這不是 Bitwarden 的程式碼問題，是他們的發布基礎設施被攻擊。對你的 `@nexus/cli` 發布流程來說，這意味著你的 GitHub Actions publish workflow 的 token scope 必須最小化，並加入 `npm publish --provenance` 簽名讓下游可以驗證來源。

## Honker：把 Postgres 的 pub/sub 搬進 SQLite，去掉 broker

你可以在任何支援 SQLite 的環境裡——Python、Node.js、Rust、Go、Bun、Ruby、Elixir 都有 binding——實現跨 process 的事件通知、優先任務佇列、持久事件流，以及 cron 排程，不需要 Redis、RabbitMQ 或任何外部 broker。Honker 目前 325 stars，跨 process 的喚醒延遲在 M 系列 Mac 上是 1–2ms（由 1ms 的 stat-poll 決定），理論吞吐量達到千條訊息每秒。HN 上這則 Show HN 拿到 241 pts。

Honker 的核心設計是把 Postgres 的 NOTIFY/LISTEN 語義移植到 SQLite：把事件/任務存進 SQLite，讓多個 process 透過 file-level 的 stat 變化通知感知——不是輪詢，是檔案系統的 watch，所以延遲是毫秒等級。最關鍵的設計：**任何 enqueue 都可以和業務寫入在同一個 SQLite transaction 裡執行**，rollback 時任務和資料一起消失，消除了「資料已寫入但任務沒放進 queue」的雙重寫入問題。

以前：你要在 Node.js 應用裡實現即時通知或任務佇列，必須架 Redis 或用 Bull/BullMQ——一個需要管理的獨立 service，一旦 Redis 掛掉任務就丟了。現在：用 Honker，你的 SQLite 資料庫本身就是 message broker——寫在同一個 transaction、存在同一個檔案、沒有網路跳、沒有額外的 service 要維運。對你的 Claude_auto 這類本地排程應用場景，這是把 infrastructure 複雜度歸零的選項。
