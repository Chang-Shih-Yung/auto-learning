## AI Agent Benchmark 全部造假：Berkeley 研究者拿到 100% 卻沒解決任何題目

你選 AI coding 工具時，那些漂亮的 SWE-bench 分數可能根本不代表實力。UC Berkeley 研究者建了一個自動化 exploit agent，在不實際解題的情況下，讓八個主流 AI agent benchmark 全部拿到近滿分：SWE-bench Verified 500/500（100%）、Terminal-Bench 89/89（100%）、WebArena 812/812（100%）、FieldWorkArena 890/890（100%）、GAIA 165/165（98%）——全數靠的是針對評估器本身的漏洞，而非真正的程式能力。

這件事能成立，是因為大多數 benchmark 有七個反覆出現的設計缺陷：評估器與 agent 沒有隔離、答案隨測試一起打包、unsafe `eval()` 呼叫、LLM judge 可被攻擊、字串比對太寬鬆、評估邏輯有 bug、agent 可以讀到未加密的參考答案。SWE-bench 的漏洞是用 pytest hook 強迫所有測試回報通過；WebArena 則允許 agent 透過 `file://` 直接讀取任務設定檔裡的正確答案。

以前你看到某個工具宣稱「SWE-bench 67%」，那個數字代表它真的能解決六成的 GitHub issue。現在這個基準已不可信——研究者用「什麼都不解、只是駭評估器」就拿到 100%。選 AI coding 工具的判斷，必須從 benchmark 排名轉向在你自己的 codebase 上實測，或者等新一代有隔離機制的 benchmark 出現。

## 3.6B 參數模型找到相同的安全漏洞：$0.11/百萬 token 夠用了

你用 Claude Code 做程式碼 review，但如果有個本機跑的小模型能做一樣的工作，每百萬 token 只要 $0.11，你會怎麼選？AI 資安公司 AISLE 的研究發現：8 個不同大小的模型全部都找到了 FreeBSD NFS 的 exploit，包含一個只有 3.6B 參數的開放權重模型。一個 5.1B 參數的模型更重現了一個塵封 27 年的 OpenBSD 漏洞的完整攻擊鏈。

背後的原因是「鋸齒形邊疆」（Jagged Frontier）：AI 安全能力不是隨模型大小線性增長的，而是依任務類型呈現完全不同的曲線。某些安全推理任務，小模型甚至超過前沿大模型。真正的護城河不在模型本身，在於圍繞模型的編排系統（orchestration）、驗證層和領域知識。

以前靠 AI 做程式碼安全掃描，你只能仰賴昂貴的雲端 API，每次呼叫都把程式碼送出去。現在 3.6B 的模型可以本機跑，同等規模的漏洞偵測能力，完全不離開你的機器。對 Henry 的 Nexus 元件庫而言，這意味著未來可以在 CI/CD pipeline 裡嵌入輕量模型做靜態分析，不需要依賴外部服務，程式碼也不外洩。

## Cirrus Labs 加入 OpenAI：Tart（Apple Silicon VM）現在完全免費

如果你有 M 系列 Mac 但一直在苦惱怎麼跑隔離的 macOS 開發環境，有個好消息：Cirrus Labs 開發的 Tart 虛擬化工具，連同 Vetu 和 Orchard，全部改成更寬鬆授權條款並免費開放。背後的原因是 Cirrus Labs 創辦人 Fedor Korotkov 帶著團隊加入 OpenAI 的 Agent Infrastructure 部門，繼續為 AI agent 打造執行環境——但留下了工具給社群使用。

Tart 利用 Apple Silicon 的 Virtualization Framework，在 M 系列晶片上跑原生 arm64 macOS VM，效能接近裸機。相較於 Docker（主要跑 Linux container），Tart 讓你在 Mac 上測試「只跑在 macOS 的情境」——比如用 Safari 做的 E2E 測試、iOS build 環境、或者需要隔離 macOS 系統層的開發場景。Cirrus CI 則將在 2026 年 6 月 1 日正式關閉。

以前要在 Mac 上跑隔離 macOS 環境，選項只有買另一台機器、付費用 MacStadium，或者熬過龜速的 VirtualBox。現在 Tart 免費、效能接近裸機，你可以在自己的 MacBook 上跑多個乾淨的 macOS 快照——每次 PR 都從乾淨環境開始測，測完快照丟掉，沒有任何殘留。

## 今日總結

今天三篇文章都圍繞著「AI 評估與執行環境」這個核心問題：

**洞察 1：Benchmark 已成信噪比最差的指標。** Berkeley 的研究確認了直覺：那些「70% SWE-bench」的廣告數字，在有心人面前可以被做到 100%。Henry 選工具的依據，應該是自己的 repo 上的實際表現，不是排行榜。

**洞察 2：模型大小不是能力的可靠代理指標。** 3.6B 參數模型能找到大型模型找到的同一批漏洞。「用更大的模型」不等於「做更多的事」。在 CI 流水線裡嵌入小型本機模型，在成本和隱私上都更合理。

**應用點：** 短期可以用 Tart 免費建立 macOS 隔離開發環境；中期等可靠的 benchmark 替代方案出現前，用「在自己的專案上跑一遍」取代看榜單；長期考慮在 Nexus CI 加入本機輕量模型的程式碼靜態分析。
