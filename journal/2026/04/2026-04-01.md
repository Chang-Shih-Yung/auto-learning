## Claude Code 原始碼洩漏：23 個情緒偵測 regex、每天浪費 250,000 次 API 呼叫，以及未發布的 KAIROS 模式

你每天用來寫程式的 Claude Code，有人用 MITM proxy 把它的 API 流量全攔截下來，反向工程了整個系統。結果：它用 **23 個正則表達式**偵測你說出「wtf」或「so frustrating」的瞬間；每天因為 autocompaction bug 白白浪費 **250,000 次 API 呼叫**（三行程式碼就能修）；還有一個叫 **KAIROS** 的未發布模式，具備記憶蒸餾、GitHub webhook 觸發、和 5 分鐘 cron 自動刷新——一個完整自主代理架構的雛形。

這些設計不是隨機的，背後都有工程取捨。挫敗感偵測用 regex 而不是 LLM，因為便宜且夠用；客戶端證明（`cch=5b52c`）用 Zig HTTP stack 在傳輸層注入 hash，確認請求來自合法 Claude Code binary——這也是 Anthropic 對 OpenCode 提告的技術依據；反蒸餾機制（`ANTI_DISTILLATION_CC`）會往 API 請求裡注入假 tool 定義，讓競爭對手的 traffic recording 訓練資料被汙染。另外還有「undercover mode」：在非 Anthropic 的 open source repo 裡，Claude 會避免提到內部代號「Capybara」「Tengu」或 「Claude Code」本身。

以前 Claude Code 是黑箱，你只知道它「能做什麼」，不知道「如何設計」。現在你能讀懂它的工程決策：為什麼用 regex 不用 LLM、為什麼需要 transport-layer DRM、KAIROS 預示著什麼樣的 autonomous agent 架構。如果你在建自己的 AI workflow 排程系統，這是罕見的第一手架構參考——KAIROS 的「5 分鐘 cron + 記憶蒸餾 + webhook 觸發」結構，正好是 Claude_auto 的進化方向。

## 1-Bit Bonsai：0.57GB 跑 4B 等級 LLM，132 tokens/sec，M4 Pro 上全速運行

你的 M4 Pro MacBook 可以用 **0.57GB** 的記憶體跑一個 4B 級別的模型，速度 **132 tokens/sec**；8B 版本只需 **1.15GB**，benchmark 跑出 70.5 分，和全精度 8B 模型打平——同樣的記憶體，以前只能跑一個勉強能用的 1B 量化模型，現在你有完整 8B 等級的推理能力。更小的 1.7B 版本跑在 iPhone 17 Pro Max 上，同樣達到 130 tokens/sec，所有推理在設備端完成。

傳統量化把每個 weight 壓縮到 4-bit 或 8-bit；1-bit 代表每個 weight 只有 `+1` 或 `-1`，整個 8B 模型壓到比一張 PNG 還小。Caltech 的研究讓這件事在不損失精度的前提下成立：設計核心從「參數量」轉為「每 bit 的智能密度（intelligence per bit）」。矩陣運算從浮點數乘法變成純粹的加法——這是速度快 **8 倍**、能耗低 **5 倍**的根本原因，而不是單純壓縮。PrismML 稱它為第一個商業可行的 1-bit LLM（有別於過去只能在學術環境跑的實驗版本）。

以前要在本地跑一個 decent LLM，你需要高端 GPU、大顯卡記憶體、或花錢叫 cloud API。現在 0.57GB 就夠，風扇不狂轉、資料不外洩、不需要訂閱。對你的 AI workflow 架構而言：本地 inference 不再是效能妥協，而是一個真實的架構選項——低 stakes 的 agent task 可以在本地跑，高 stakes 的才送到 Anthropic API，路由成本驟降。

## Slop 不是未來：token 經濟學會逼出好程式碼

你作為同時用 AI 工具寫程式、又在維護 Nexus 元件庫的開發者，現在有一個更精準的框架來評估 AI 生成的程式碼品質：問「這段程式碼需要多少 context tokens，才能被 AI 正確理解和修改？」AI 普及後每個開發者的程式碼產出從 **4,450 行跳到 7,839 行**（+76%），PR 大小中位數增加 **33%**（57 → 76 行），軟體停機事件自 2022 年持續上升。這不是巧合。

Greptile 的論點是：複雜程式碼讓每次 AI 修改都更貴——需要更多 context tokens 來理解、更多 tokens 來生成正確的改動、更容易出錯就要更多 tokens 來修復。AI 公司按 token 計費，因此市場有直接財務誘因讓模型產出更簡潔的程式碼。「好程式碼」在 AI 時代從工程哲學問題，變成 economics 問題：**token 效率高的程式碼，就是好程式碼。**

以前「好不好」靠 code review 文化和個人修養——很難量化、難以說服 PM 值得投入。現在你有一個可量化的說法：「這段程式碼的 context footprint 是 X tokens，修改它的成本是 Y tokens」。對你的 Nexus 設計系統而言，這是新的評估維度——每個元件的 token footprint 越小，AI copilot 的修改成本越低、正確率越高。你在 EQS 和 MES 都踩過 legacy code 累積成指數維護負擔的坑；token 經濟學只是讓這件事變得可見、可計算。

## 今日總結

**三個核心洞察：**

1. **工具透明度 = 架構學習機會**：Claude Code Source Leak 讓一個日常工具變成可研究的系統設計案例。KAIROS 的 cron + webhook + 記憶蒸餾架構，正好是下一版 Claude_auto 的進化藍圖。工程取捨（regex vs LLM、transport-layer DRM）都有可移植的設計邏輯。

2. **本地 inference 門檻崩潰**：1-Bit Bonsai 讓 8B 等級模型的記憶體需求從 ~16GB 降到 1.15GB，讓「本地 vs 雲端 routing」成為實際的架構決策，而不只是理論選項。Henry 的 agent pipeline 可以開始考慮 task routing 設計。

3. **Token 效率是 AI 時代的程式碼品質新語言**：「複雜程式碼在 AI 時代更貴」不只是哲學，是財務邏輯。Nexus 元件庫的設計可以加入 token footprint 作為評估維度，讓 AI-first 設計系統有可量化的品質指標。
