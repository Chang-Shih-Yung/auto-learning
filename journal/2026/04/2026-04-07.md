# 2026-04-07 學習日誌

## 1. GitHub Issue #42796：有人分析 17,871 個 thinking block，找出了 Claude Code 退步的原因

自從 2026 年 2 月 Anthropic 推出「thinking content redaction」更新（`redact-thinking-2026-02-12`），社群一直反映 Claude Code 在複雜工程任務上明顯退步。這份 GitHub Issue 不是一般抱怨帖——作者分析了 6,852 個 Claude Code session 檔案，提取了 17,871 個 thinking block 和 234,760 個 tool call，給出了數據支撐的結論：extended thinking tokens 對複雜多步驟任務是「結構上必要的」，不是「可有可無的附加功能」。Thinking redaction 後，模型的 tool usage pattern 從「先研究再修改」（research-first）轉向「直接修改」（edit-first），這就是為什麼你會看到 Claude 在不充分理解 context 的情況下直接改程式碼。

Thinking content redaction 的原因是 Anthropic 想減少回傳的 token 量（節省用戶 API 費用 + 提升速度）。但 extended thinking 的功能不只是「讓你看到 Claude 在想什麼」——它是模型在複雜任務裡進行多步推理的工作空間。把它壓縮掉，等於把草稿紙從工程師手裡拿走，讓他直接在正式文件上工作。Session 越長、任務越複雜，影響越大。

對你的 Claude Code 排程任務，這個分析有兩個直接應用：一是如果你在某個複雜任務（多檔修改、架構重構）上遇到奇怪的行為，先考慮是不是 thinking depth 不足；二是社群已經整理出幾個繞道方法——在 CLAUDE.md 裡加入明確的「先研究後修改」步驟指示，或把大任務拆成「先分析 → 再執行」的兩階段 session，人工提供 thinking 空間。CLAUDE.md 裡加入「You have permission to edit any file. Do not ask for confirmation.」這樣的明確授權，可以減少約 70% 的多餘確認中斷。

## 2. Anthropic 年收入突破 300 億美元，鎖定 350 萬個 Google TPU 應對算力需求

Anthropic 與 Google 和 Broadcom 宣布擴大計算合作：從 2027 年起，Anthropic 將獲得約 3.5 GW 的 TPU 算力，對應近 100 萬個 Google TPU v7p 單元。這個數字的背景是：Anthropic 目前年營收已超過 300 億美元（2025 年底約 90 億美元，不到 18 個月內增長超過 3 倍）。每年付 Anthropic 超過 100 萬美元的企業客戶超過 1000 家，且這個數字在不到兩個月內就翻了一倍。也就是說，Claude 的算力擴張不是賭注，是已知需求的規模跟上。

Google TPU v7p 是 Google 自研的 AI 加速晶片，由 Broadcom 設計。這次合作的結構是：Broadcom 為 Google 設計和供貨下一代 TPU 和 AI 資料中心網路設備（合約到 2031 年），Google 再把這些算力透過租賃方式提供給 Anthropic。Anthropic 過去已承諾 210 億美元的晶片訂單——這次 3.5 GW 是在此基礎上的進一步擴大，代表未來的 Claude 模型訓練和推論基礎設施已提前鎖定到幾年後。

對你的日常使用，這有幾個層面的意義：一是 Claude API 的長期可用性和穩定性有了算力保障，排程任務不用擔心「Claude 因為算力不足而回應變慢」；二是 300 億美元年收入意味著 Anthropic 有足夠的資源繼續大規模的安全研究和能力提升——你在這份日誌裡追蹤的各種 Claude 改進，背後都有穩定的研發投入在支撐。

## 今日總結

今天兩篇文章共同揭示了 Claude 生態系的一個重要張力：**能力需求 vs 算力成本**。

- **Issue #42796** 說：為了省 token、省錢，Anthropic 削減了 thinking tokens，結果讓複雜任務變難。這是一個真實的工程 trade-off，不是疏失——只是需要使用者端做對應的調整（research-first 指示、分階段 session）。
- **算力擴張**說：Anthropic 的收入成長比算力擴張快，所以他們在大量採購未來算力——3.5 GW 的 TPU 儲備是為了跑得上來。

兩篇合起來的洞察：現在的限制（thinking redaction、rate limit）是暫時的供需失衡，不是能力退步。你的應對策略是在現有限制下設計最有效率的工作流，等算力跟上，這些限制會自然鬆綁。
