## Claude Code v2.1.105 — EnterWorktree 路徑切換 + 串流自動重試

你的 Claude Code 現在可以直接跳進「已存在的 worktree」——不用每次從頭建立新的，只要傳入 `path` 參數就能讓 Agent 切換到任何既有分支工作區。更重要的是：API 串流現在內建 5 分鐘 timeout 加自動重試，長任務跑到一半網路不穩不再代表全部重來。`/doctor` 指令也重新設計，狀態圖示一眼看懂；技能描述上限從 250 字元擴到 1,536，自訂技能終於可以把完整的使用說明寫進去。

這背後的技術改動集中在兩處。第一：`EnterWorktree` 工具新增 `path` 參數，Agent 可以從 filesystem 路徑直接定位已有的 worktree，而不只是建立新的隔離分支。第二：串流層加入真正的 timeout + retry 機制——以前 API 串流中斷只會丟出錯誤訊息，現在是在基礎設施層自動吸收短暫的網路波動，再重新接上。

以前：長任務跑 30 分鐘，網路抖一下，整個 session 報廢，只能從頭再來。現在：自動重試，任務繼續；worktree 不用每次新建，直接給路徑，Agent 知道去哪裡。對每天在 Claude Code 裡跑多個分支任務的工作流來說，這兩個改動加在一起省掉的等待時間相當可觀。

## GitHub Stacked PRs — 大型 PR 原生拆解，自動 Rebase

你現在可以把一個大型功能（例如 Nexus 某個複雜元件的重構）拆成 3 到 5 個相互依賴的小 PR，每個獨立送審、獨立合併——GitHub 原生處理整個鏈的 rebase，不需要你手動維護分支順序。當某一層 PR 合併之後，下面的 PR 自動重新 rebase 到正確的 base，stack 繼續保持乾淨狀態。支援 `gh stack` CLI 操作本機工作流，也支援 AI coding agent 直接操作 stack。

每個 PR 的 base branch 指向鏈結中前一個 PR 的分支，最底層才真正合向 main。GitHub 在合併事件發生時自動觸發 rebase 流程，保護規則（branch protection rules）統一對最終 target branch 生效，而不是對 stack 中間的每一層單獨生效。

以前：大型功能要麼一次整包 merge（reviewer 看不完）、要麼手動維護一堆依賴分支（rebase 地獄）。現在：`gh stack` 一行指令建立整個 PR 鏈，合併一層、下一層自動就位。目前功能在 private preview 階段，但 CLI 工具已可透過 `npx skills add github/gh-stack` 安裝。

## AI Agent 記憶體的真正問題：不是找不到，是連不上

你在設計 AI agent 的記憶架構時，最常想的問題是「怎麼讓它找到正確的記憶」——但 500 個實驗告訴你，這個假設本身就是錯的。實驗橫跨 250 個任務、5 種類型，有記憶系統的 agent 整體勝率是 61.6%，但在「最難的路由任務（Track C）」這一類，rubric delta 高達 +0.40，而在其他類型幾乎沒有優勢。原因是：20% 的任務嘗試從 skill memory 提取技能，但 0 個任務真正用到——因為找到的全是空殼模板「implement [target]」，語義相似度只有 0.02。

問題不是 recall（檢索），是 binding（綁定）。Recall 是把記憶取出；Binding 是把程序記憶（你知道怎麼做）、情節記憶（你上次怎麼做的）、因果推理（為什麼這樣做）、領域事實（這個領域的規則），打包成一個 Memory Bundle 同時載入——而不是四個各自優化的獨立系統，互相之間沒有連結。這個系統的失敗模式是：兩套完整的記憶架構存在同一個 codebase 裡，但零整合，各說各話。

以前的設計假設：強化 embedding model、提升向量搜索精度，recall 準確就是好記憶。現在的發現：即使 recall 準確，如果程序記憶和情節記憶是分開存取的，agent 還是會在任務執行時丟失上下文，重複犯錯。真正有效的記憶系統要讓「知道怎麼做」和「上次做了什麼」在同一個 bundle 裡一起出現。

## Cloudflare `cf` CLI — 3,000 個 API 操作，不到 1,000 Tokens

你現在可以用一個統一的 CLI 工具（`cf`）操作 Cloudflare 全平台 100+ 產品、近 3,000 個 API 端點，包括 KV、R2、D1、Durable Objects。本機開發時按下鍵盤 `e`，Local Explorer 直接打開讓你預覽本地的 KV 和 D1 資料，不需要部署到雲端再回頭看結果。對 AI agent 來說更重要的是：整個 Cloudflare API 被壓縮到不到 1,000 tokens 的 MCP Server 格式，agent 可以在 context 預算不爆的前提下理解整個平台。

這套 CLI 從 TypeScript 型別架構自動生成，而不是人工維護文件。這個設計決定帶來的核心效果是強制命名一致性：永遠用 `get`（不用 `info`）、統一 flag 名稱（`--force`、`--json`）、清楚標記操作的是本機還是遠端資源。一致性不靠 code review 維持，直接在 schema 層強制執行。

以前：Cloudflare 100+ 個產品各自有 CLI、SDK、REST API，命名規則不統一，AI agent 呼叫時需要為每個產品記住不同的 subcommand 格式，出錯率高。現在：一套統一 schema 生成所有指令，< 1,000 tokens 就能讓 agent 掌握完整的 API surface，跨產品操作不需要猜命名。這個壓縮比（3,000 個操作 → < 1,000 tokens）本身就是 MCP Server 設計的重要參考指標。

## 今日總結

今天四篇文章圍繞一個共同主題：**讓 AI agent 可靠地操作複雜系統**。

- Claude Code v2.1.105 的 EnterWorktree path 和串流重試，解決的是 agent 在長任務中的**環境穩定性**問題
- GitHub Stacked PRs 解決的是大型功能在 CI/CD 鏈上的**合併可靠性**問題
- Agent Memory Binding 研究揭示的是記憶系統的**整合缺口**——找得到不代表用得上
- Cloudflare cf CLI 示範的是如何把 3,000 個 API 壓縮成 agent 可靠理解的**最小接口**

這四個方向合在一起，勾勒出 2026 年 AI agent 工程的核心挑戰：不是讓 AI 更聰明，是讓它在真實的、有噪音的、有複雜依賴的系統中**可預測地運作**。Henry 的 Nexus + Claude_auto 工作流正好處在這個問題的核心——每一篇都有直接可以實驗的切入點。
