# 2026-04-23 學習日誌

## Claude Code v2.1.118 — Hooks 直接呼叫 MCP 工具

今天起，你在 Claude Code 的 hooks 裡可以直接寫 `type: "mcp_tool"`，讓 pre-edit、post-edit、pre-commit 等任何 lifecycle event 直接呼叫你本地的 MCP Server，不再需要透過 shell 命令轉一層。同一版本還帶來：`/usage` 整合了 `/cost` 和 `/stats` 兩個舊命令（兩者仍可作為快捷鍵使用）；`/theme` 可以建立並切換命名主題，也能直接手動編輯 `~/.claude/themes/` 裡的 JSON；Vim visual mode（`v`）和 visual-line mode（`V`）正式加入，支援選取、操作符與視覺回饋；`DISABLE_UPDATES` 環境變數讓你完全鎖定所有更新路徑，包括手動 `claude update`。

hooks 能呼叫 MCP 工具，代表 Claude Code 的自動化邊界往前推了一格。以前 hook 只能跑 shell 命令，要觸發 MCP 工具必須透過 subprocess 或外部腳本繞路；現在 hook 可以直接把 MCP 工具當作一等公民呼叫，Claude Code 的 lifecycle 和你的 MCP 工具鏈之間不再有隔閡。plugin 也可以透過 `themes/` 目錄打包主題一起發布，整個工具生態的整合度再拉高一層。

以前要讓「儲存檔案後自動觸發 Figma token 同步」，你得在 hook 裡寫一個呼叫 MCP Server 的 shell 腳本，還要處理路徑、權限和錯誤輸出。現在直接在 hooks 設定裡宣告 `type: "mcp_tool"` 加上工具名稱就搞定，整條流水線住在設定檔裡，可以 commit、可以 review，不再是散落在各處的膠水腳本。

## LLM 過度編輯：你叫它改一行，它重寫了整個函式

研究團隊用 400 個來自 BigCodeBench 的問題測試主流 LLM 的編輯行為：每個問題都是把程式碼「精確破壞一個地方」，正確答案就是把那一個地方改回來，不多不少。結果呢——GPT-5.4 把 `range(len(x) - 1)` → `range(len(x))` 的一行修改，改成了整個函式重寫：加入 `None` 檢查、`np.asarray` 轉換、有限值遮罩、陣列大小驗證，全部不在原始需求裡。用 Levenshtein Distance 量化，所有受測模型都呈現不同程度的過度編輯。解法出乎意料地簡單：在 prompt 裡明確寫「只改最少必要的地方」，每個模型的距離分數都改善，Pass@1 也跟著提升。

這個問題叫 over-editing：模型的輸出在功能上正確，但結構上偏離原始程式碼的幅度超過最小必要範圍。研究的設計很聰明——因為「正確修改」是程式化製造的，所以 ground truth 是客觀的，不需要人工判斷。問題根源可能是模型的訓練偏好「全面整潔的程式碼」而非「局部精準的修改」，或者對 implicit quality requirements 過度響應。

以前你叫 Claude 「修一個 bug」，結果整個函式邏輯都被重組，RxJS operator 鏈換了順序，CSS 命名也順手改了。你以為 AI 很聰明，其實它只是太勤勞。現在你知道了：在 prompt 裡加上「只改最少必要的地方，不要重寫不相關的程式碼」，模型的行為就會收斂。對於精心設計的 Angular 元件和 RxJS 數據流，這一行 prompt 的投資報酬率極高。

## Show HN Design Slop：500 頁面分析，AI 正在讓網路設計同質化

一位開發者對 500 個 Show HN 頁面做了系統性分析，建立了 15 個 AI 設計模式清單，並計算每個頁面觸發幾個。觸發 5 個以上 = 「重度 slop」。排名前幾的信號：彩色左邊框（「幾乎跟 em-dash 一樣可靠的 AI 生成信號」）、頂部有圖示的功能卡片、漸層背景搭配毛玻璃卡片。有趣的背景：自從 Claude Code 普及，Show HN 的投稿量暴增三倍，HN 版主已被迫限制新帳號的 Show HN 投稿資格——所謂的「民主化開發」正在催生大量外觀相同的產品。

AI 設計 slop 的成因在於工具鏈的收斂：大多數人用 v0/Bolt/Lovable 生成 landing page，這些工具的預設輸出高度相似，且都走向「安全的現代感」——漸層、圓角、毛玻璃、統一的排版節奏。因為沒有設計系統作為錨點，AI 就用「看起來最像科技產品」的模版填充，而所有模型訓練資料裡「看起來像科技產品」的長相是高度重疊的。

以前看到一個 Show HN，你憑直覺感覺「這個很 AI」但說不清楚為什麼。現在有了量化框架：這個頁面觸發了哪幾個模式？超過 5 個就是結構性 slop，不是個人風格。更重要的是，這個清單可以倒過來用：在建 Nexus 元件庫時，把這 15 個模式當作禁用列表，確保你生成的元件有設計系統的識別度，而不是 shadcn 預設值的複製品。

## Qwen3.6-27B：27B 參數打趴前代 397B 旗艦，跑在本地

Alibaba 的 Qwen 團隊釋出 Qwen3.6-27B：27B 參數的 dense 模型，在所有主要 coding benchmark 上超越前代旗艦 Qwen3.5-397B-A17B，體積縮小到 1/14（55.6GB vs 807GB）。Q4_K_M GGUF 量化版只要 16.8GB，用 `llama.cpp` 的 `llama-server` 就能跑，context window 65,536 tokens，實測速度約 25 tok/s——在本地硬體上，20,000 字的程式碼庫可以一次送進 context，幾秒內就有回應。模型已上架 Hugging Face，和 ModelScope。

技術突破點在於訓練效率的提升——同樣的 benchmark 分數，所需的參數量從 397B 降到 27B，意味著訓練和推理的計算量都大幅下降。這類「效率跳躍」通常來自架構改進（attention 機制優化、更好的 tokenizer）加上更高品質的訓練資料集；Qwen 團隊的 QwenClawBench 和 NL2Repo benchmark 設計也專門針對 agentic coding 場景，而不只是靜態程式碼補全。

以前要跑旗艦級 coding 模型，你需要 800GB+ 的 VRAM 集群，或者付 API 費用讓資料上雲端。現在 16.8GB GGUF 可以直接在有 32GB RAM 的 Mac 上跑，速度 25 tok/s 夠做即時程式碼輔助——對於需要保密的 Nexus 商業邏輯，或者網路不穩的環境，這是一個真正可用的本地備援方案。
