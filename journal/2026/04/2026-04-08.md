### 1. Anthropic Project Glasswing：AI 第一次在弱點掃描上超越頂尖人類

Anthropic 今天宣布 Project Glasswing，和 AWS、Apple、Google、Microsoft、Cisco、NVIDIA 等 11 家科技公司合作，把最新的 Claude Mythos Preview 部署在零時差弱點掃描任務上。這個模型在 CyberGym 弱點重現測試中拿到 83.1%，比 Claude Opus 4.6 的 66.6% 高出 16.5 個百分點；SWE-bench Pro 達到 77.8%、Terminal-Bench 2.0 達到 82.0%。實測結果：已在 OpenBSD（27 年歷史）、FFmpeg（16 年）、Linux kernel 裡找到數千個 zero-day 漏洞，而且是在沒有人類引導的情況下完成的。

Mythos Preview 的核心突破是把「找漏洞」和「修漏洞」放進同一個 adversarial loop：模型先從 offensive 角度識別可被利用的記憶體問題、輸入驗證缺口、加密實作錯誤，再切換到 defensive 模式自動生成 patch 或 mitigation。這之所以現在可行，是因為 Mythos Preview 的 coding 能力已到達「在大多數任務上超越大多數人類開發者」的門檻——Anthropic 自己在公告裡寫道：「AI models have reached a level of coding capability where they can surpass all but the most skilled humans at finding and exploiting software vulnerabilities。」

以前發現 zero-day 漏洞需要頂尖資安研究員手動審查程式碼，數週到數個月才有結果，而且這種能力極度稀缺。現在 Mythos Preview 幾小時內就能掃完 27 年歷史的 OpenBSD 程式碼庫。對正在開發 MCP Server 和 CLI 工具的開發者來說，AI code review 的可信度即將達到「能自動發現你寫的工具裡的安全問題」的門檻——攻防兩側的能力同時提升。

### 2. GLM-5.1：754B 本地模型，能力接近 Opus 4.6，但超過 100k token 就壞掉

z.ai 推出 GLM-5.1，一個 754B 參數的開放大型語言模型，HN 社群回報它在複雜後端任務上「比 Claude 和 Codex 更能遵循深層執行計畫」，部分測試者認為特定 coding 任務的能力與 Claude Opus 4.6 相當。對想在本地跑大模型的開發者來說，量化版本（IQ4_XS）壓縮到約 361 GB，可以在高端 workstation 或多 GPU 的本地伺服器上部署，資料完全不離開自己的機器。

GLM-5.1 的長處是 instruction-following 深度——即使在計畫執行的後期階段、context 已累積了大量對話歷史，模型仍能準確跟隨複雜規格，不會「忘記」早期的約束條件。這對需要長時間執行的 AI agent 任務理論上很重要。不過 HN 討論也揭露了一個明確的上限：超過 100k–128k tokens 後，模型出現「context rot」，回應品質急劇下滑，甚至變成亂碼，需要手動 compact 或重開 session 才能恢復。

以前本地模型的 coding 能力距離 Claude Opus 有很大的差距，要在資料敏感環境（企業內網、不能把程式碼傳到雲端）用到同等級的 AI 輔助幾乎不可能。GLM-5.1 第一次提供了一個可以認真評估的替代方案。但 100k token 的 context rot 限制說明它目前還不適合長時間執行的 AI agent 工作流——更像是「強力的 one-shot 助手」而不是「能跑一整天的自主 agent」。

### 3. Railway 離開 Next.js：build 從 10 分鐘縮到 2 分鐘，靠的是 Vite + TanStack Router

Railway 今天發布了一篇遷移報告：他們把前端從 Next.js 完整遷移到 Vite + TanStack Router，CI build 時間從超過 10 分鐘縮短到 2 分鐘以內，dev server 啟動從數十秒變成即時（0 秒等待）。光是移除 Next.js 這一層就省掉 6 分鐘，其中有整整 3 分鐘卡在「finalizing page optimization」——一個和 Railway 的產品完全無關的 build 步驟。如果你的 CI 每天跑 20 次，這等於每天省下 100 分鐘的開發者等待時間。

Next.js 的效能開銷來自它的設計定位：server-first 框架會在 build 時對每個路由做 SSR 最佳化分析和靜態生成，即使那個路由根本沒有 server-side 邏輯。Railway 的 app 是重度 client-side 的即時介面，大量用 WebSocket 推送狀態，幾乎沒有 SEO 需求——這種架構和 Next.js 的設計假設完全相反，每次 build 都在為從來沒有用到的功能買單。

以前「前端框架選 Next.js」幾乎是不需要解釋的預設答案，選手 Vercel 還幫你做 CI 最佳化。現在 Railway 的案例說清楚了邊界：Next.js 適合有 SEO 需求、以 SSR 或 SSG 為主、MDX 靜態生成的網站；但如果你的 app 是以 client-side 狀態和即時資料流為主，Vite + TanStack Router 讓 build 時間少 80%，dev server 是真正的即時啟動。Claude_auto 這個部落格用 Next.js 是合理的選擇（SEO + MDX），但 Nexus demo app 或未來的即時 dashboard 工具選框架之前，這篇值得先讀一遍。

## 今日總結

今天三篇文章從不同角度描述了一個共同的訊號：**AI 的能力門檻正在快速移動，但「適合哪個場景」的判斷比「最強的工具」更重要**。

Glasswing 代表 Claude Mythos 在 coding 能力上進入了新的量級（83.1% CyberGym、77.8% SWE-bench Pro），這個提升不只是「寫程式更好」，而是「能找到人類找不到的安全問題」——意味著 CI/CD 裡的 AI security audit 在技術上變得可行。GLM-5.1 則顯示本地大模型第一次接近 frontier 水準，但 100k token context rot 暴露了它的邊界：對 one-shot 任務強，對長 agent session 不可靠。Railway 的 Next.js 遷移案例是一個清醒劑：再好的框架也有不適合的場景，選錯的代價是每天多等幾百分鐘的 build time。

**本週的核心洞察：** 工具選擇的標準不是「這個工具有多強」，而是「這個工具的設計假設是否符合你的場景假設」——Next.js vs Vite 如此，Claude API vs 本地 GLM 如此，雲端 security scan vs 本地 CI check 亦然。
