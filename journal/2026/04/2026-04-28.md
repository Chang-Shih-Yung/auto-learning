## Claude Code v2.1.121：MCP `alwaysLoad` 解開 deferred loading 的最後一個結

你可以在 `settings.json` 裡為某個 MCP server 加上 `alwaysLoad: true`，把那組工具固定保留在 prompt 裡，不被 v2.1.116 引入的 tool-search deferral 機制延後載入。如果你在 Nexus 用了一個 design-token MCP server，每次都要手動 `mcp__searchTool` 才能調出來——現在加一行設定，這組工具就跟原生 Read / Edit 一樣每次都直接可用。同一版還補了 `claude plugin prune` 清掉 marketplace 移除後的孤兒依賴、`/skills` 支援邊打邊濾、scrollable 對話框可用方向鍵與滾輪、PostToolUse hook 可以對任何工具的輸出做覆寫；同時修掉 image processing 與 `/usage` 的記憶體洩漏，以及工作目錄被刪掉之後 Bash 工具就再也不能用的長期 bug。

deferred loading（v2.1.116, 2026-04-21）的設計假設是「冷工具 → 暫時不顯示，省 token」，但有些工具是熱工具——Henry 的 design token sync、Nexus CLI、Figma MCP——每一個 session 都會用到，每次 deferral 反而多一輪 search 往返。`alwaysLoad` 等於是給 server 維護者一個聲明式 hint：「這些工具值得 token 成本，不要省」。PostToolUse 的 output replacement 是另一條軸線：以前 hooks 只能 block / allow，現在可以重寫工具回傳給模型的內容，等於你能在 hook 層做 sanitize、redact、format normalization，把模型不該看到的雜訊在 hook 過濾掉。

以前你只有兩個極端：所有工具一次載入（吃光 context），或全部 deferred（熱工具每次多繞一圈）。現在多了一格中間光譜——以 server 為單位設定 alwaysLoad，把熱工具留下、冷工具讓系統自動延後。這跟 4 月 21 日 ggsql 把計算推到資料庫端、4 月 27 日 Cloudflare Code Mode 把 schema 推到 execution time 是同一條線索：複雜度不是消失，是被搬到正確的位置；而 v2.1.121 把這個搬移的決策權交還給開發者。

## Dirac：用 AST + Hash-Anchored edits 把 AI coding agent 的單次成本壓到 0.18 美元

你可以用 `npm install -g dirac-cli` 裝這個 OSS coding agent，跑同一批 8 個 refactor 任務，平均每個任務只花 0.18 美元——同樣的任務換成主流閉源工具平均要 0.49 美元，省了 64.8%。在 Terminal-Bench 2.0 上，Dirac 用 `gemini-3-flash-preview` 跑出 65.2% 通過率，超過 Google 自家基線的 47.6%、也勝過閉源工具的 64.3%。它支援 Anthropic Claude、OpenAI、Gemini、Groq、Mistral、x.ai、HuggingFace 等只要支援 native tool calling 的模型，不走 MCP 路線。

省成本的祕密在三個技術選擇。Hash-Anchored edits 把要修改的程式碼用 AST 的穩定 hash 而非行號錨定，模型輸出的 patch 即使檔案在編輯期間被別的工具動過，hash 對得上就能精準命中——失敗重試和 token 浪費直接消失。AST-Native precision 讓 agent 看 TypeScript / Python / C++ 的時候不是看純文字，是看語法樹，function extract 不需要丟整檔上 context。Multi-File Batching 把多個檔案的編輯合併到單次 LLM 呼叫，省掉 round-trip 的 token 重複開銷。整套設計刻意避開 MCP，因為 MCP 的工具清單在他們的成本模型裡，本身就是個 token 黑洞。

以前要選 AI coding agent，方向只有兩條：閉源精修（Cursor / Copilot / Claude Code），或 OSS 但跑分輸給 SaaS（早期 OpenDevin / Cline）。Dirac 在開源 + 跑分贏 + 成本砍 1/3 三個座標同時押注，等於把「AI coding 一定要付訂閱」這個假設推翻一半——它讓你看到 MCP 不是唯一答案，AST + 編輯協定也是一條路。對 Henry 這種同時在用 Claude Code 和自己寫 MCP server 的人，Dirac 是一個非 MCP 路線的對照組，看清楚每個技術選擇的代價。

## Browser Harness：592 行 Python 寫一個能自己改自己的 LLM 瀏覽器 agent

你可以用一個約 592 行的 Python 專案，讓 LLM 直接接 Chrome DevTools Protocol（CDP）的 WebSocket，跑任何瀏覽器任務——不是預先寫死一堆 action，是讓 agent 在跑的過程中發現「我需要一個 upload_file()」就自己改 `helpers.py`、把缺的函式補上去再繼續執行。整個 codebase 拆成 `run.py`（約 36 行）、`helpers.py`（約 195 行）、`admin.py + daemon.py`（約 361 行）四個檔案，目前 GitHub 7,596 stars，Apache 風格 MIT 授權，HN 上 131 分 63 條討論。

self-healing 的核心不是「框架預先處理錯誤」，剛好相反：框架刻意做到極薄、極少抽象，把所有「該怎麼處理 PDF download / iframe / shadow DOM」的判斷留給 LLM 在 runtime 寫程式碼解決。它的單一 WebSocket 直連 CDP，沒有 Playwright / Selenium 那套 abstraction layer，模型看到的是最原始的 DOM 與 CDP 事件——同時意味著模型必須自己知道如何點擊、如何等待，但也意味著沒有任何邊界限制它的操作。「self-healing」不是框架幫你救急，是模型有寫 helper 的權限。

以前要做 LLM 控制瀏覽器，你要先選一個自動化框架（Playwright / Puppeteer / browser-use）、再寫 wrapper 讓 LLM 能呼叫它的 API、再處理 wrapper 沒涵蓋的 edge case。這套設計的賭注是：與其讓框架預測所有失敗模式，不如把那 195 行 helpers 開放給 LLM 自己改——當你的瀏覽器 agent 失敗，它的解法是寫一個新的函式而不是回報錯誤。對 Henry 這種做過 100+ 表單流程卡控、看過 selector 在生產環境破裂的人，這個設計直接挑戰「自動化要嚴謹封裝」的工程直覺。

## 今日總結

今天三條線索其實在講同一件事：**把決策權搬到正確的位置**。

- **Claude Code v2.1.121** 的 `alwaysLoad` 把工具載入決策從系統還給 server 維護者——熱工具該不該 defer，server 自己最清楚。
- **Dirac** 的 Hash-Anchored edits + AST batching 把編輯精準度從「文字行號」搬到「語法樹節點」——decision point 在 AST 層而不是 token 層。
- **Browser Harness** 的 self-healing 把錯誤處理從框架還給 LLM——能不能解這個 edge case，模型 runtime 知道得比框架設計者多。

把這三個放在一起對照 4 月 21 日的洞察「把複雜度推到正確的層」、4 月 27 日「context window 是用來做決策的，不是存歷史」，今天的延伸是：**決策權的歸屬同樣需要設計**。框架不是抽象越多越好，是要把「誰最知道這件事」的判斷搬到能做這個判斷的人或角色身上——可能是 server 維護者、可能是 LLM、可能是 AST 解析器。

對 Henry 來說，下一步的實作問題是：Nexus 的 MCP server 該如何分層 hot / cold 工具？Figma → CSS Token migration 腳本能不能改用 AST hash anchored 而不是檔名定位？這兩個是今天就能動手的小實驗。
