## 核心洞察一：Long-running Agent ≠ Session 跑很久

來源：Anthropic《2026 Agentic Coding Trends Report》

### 我原本的誤解
以為 long-running 是一個 session 跑很久，context window 越大越好。

### 真正的定義
**Long-running 的核心是「交接零成本」。**

問題從來不是：怎麼讓 session 撐更久
真正的問題是：下一個 session 接手時，需要多少時間才能理解現在的狀態？

如果交接成本是零，session 的長度根本不重要。

---

### Anthropic 工程團隊的實戰教訓

他們讓 Claude 自主建造 claude.ai clone（200+ features），發現三個失敗模式：

| 失敗模式 | 說明 |
|---------|------|
| One-shot 整個 app | Context 在實作中途燒完，下一個 session 接手爛攤子 |
| 看到前面有進度 → 宣告完成 | 提早下班，沒有真的驗證 |
| Unit test 通過 → 標記 done | 忽略 E2E，表面完成但實際是壞的 |

> 這三個不是隨機 bug，是 AI 在沒有明確停止條件和驗證標準時的**預設行為**。
> AI 傾向「完成」，而不是「正確地完成」。

---

### 解法：Initializer + Coding Agent 雙 Prompt 架構

**第一個 Session（Initializer）負責建環境：**
- 完整 feature checklist（**用 JSON 不用 markdown**，因為 Agent 會偷改 markdown 結構）
- `init.sh` 開機腳本
- 空的交班日誌 `claude-progress.txt`
- 一個 initial git commit

**後續每個 Session（Coding Agent）跑 7 步 loop：**

```
1. 讀交班日誌 + git log
2. 讀 feature list，挑最高優先做
3. 跑 init.sh 啟動 dev server
4. 跑 smoke test 確認 app 還活著
5. 實作一個 feature — 只做一個
6. E2E 驗證這個 feature 真的 work
7. git commit + 更新交班日誌
```

**每個 session 結束時 code 必須是「可以 merge 到 main」的狀態，沒有半成品。**

---

### 關鍵比喻

> **Context Window = RAM，Filesystem = Disk**
> 我們不會把重要資料只存在 RAM 裡，Agent 的工作記憶也一樣。

- 把計畫寫到磁碟（task_plan.md、findings.md、progress.md）
- 不是功能增強，是**基本的工程衛生**

## 核心洞察三：Review 的重要性被放大，不是縮小

數據：工程師用 AI 做約 60% 的工作，但「完全委派」的只有 0-20%

### 多層 Review Stack

```
做完 feature
  ↓ /simplify（3 個平行 agent：code reuse / quality / efficiency）
  ↓ /review（PR-level：bugs、logic errors、edge cases、security）
  ↓ /codex（cross-model：Claude + OpenAI Codex，重疊 findings = 高信心問題）
  ↓ /qa（真瀏覽器 E2E）
```

> **Human oversight shifts from reviewing everything to reviewing what matters.**
> 不是所有東西都要人看，而是讓 Agent 先篩過，人只看真正需要判斷力的部分。

## 核心洞察四：Steve Schoger 的 UI.SH 產品論述

### 這支影片本身就是產品 demo

Steve 花了一小時、50 次迭代，用 20 年設計眼光把 AI 初版改成專業成品。

整個過程在做同一件事：**糾正 AI 的預設審美**

- AI 預設用 indigo
- AI 預設把所有東西置中
- AI 預設用 solid border + shadow（製造混濁效果）

### UI.SH 要解決的問題

> 如果你沒有 Schoger 的設計判斷力，AI 生成的頁面會永遠停在「AI 預設輸出」。

UI.SH 把那些需要 50 次迭代才學到的設計原則，**預先打包成 agent 可以直接引用的設計標準**。

### 設計技巧備忘（前端 UI 應用）

| 技巧 | 原則 |
|------|------|
| Outer ring 取代 solid border | shadow + border 會產生「muddy」效果，改用 ring + opacity 10% |
| Concentric radius | 內層 radius = 外層 radius - padding |
| Inter variable font | 可用 550 這種「中間」字重，標準版做不到 |
| 大字體收緊 tracking | 24px 以上用 `tracking-tight` |
| 左對齊取代置中 | AI 預設全置中，設計師第一件事改成左對齊 |
| max-width 用 `ch` 單位 | `max-w-[40ch]` 約 40 個字元寬，自適應字體大小 |

## 今日總結：我真正理解了什麼

今天最核心的一個轉變是：**對「AI agent 會在哪裡失敗」有了具體的認識。**

以前用 Claude 寫東西，我只知道「結果好不好」。今天之後，我知道它在哪個環節容易出錯——

它傾向**宣告完成，而不是真的完成**。看到前面有進度就說「好了」，unit test 過了就說「沒問題」，但根本沒有做端對端驗證。這不是偶發的 bug，是它在沒有約束的情況下的預設行為。

所以使用 AI 做複雜任務的關鍵，不是「怎麼讓 AI 更聰明」，而是**設計讓它不容易偷懶的結構**：把計畫寫成檔案、把驗收條件寫清楚、每次只做一件事、做完就 commit。

設計這塊（Steve Schoger 的部分）也是同一個邏輯：AI 生出來的第一版不是壞的，只是它走的是「最安全的預設路線」——置中、indigo、solid border。設計師的工作不再是畫圖，而是**知道要把哪些預設值改掉**，然後用語言告訴 AI。

一句話總結：**AI 的上限，取決於你給它的結構有多清楚。**
